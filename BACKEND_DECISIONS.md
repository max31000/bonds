## Решение 1: Dapper не использует кастомный ITypeHandler для enum-свойств на запись

**План не указывал** на этот риск явно.

**Решение:** Dapper, строя параметры INSERT/UPDATE из POCO/анонимного объекта, для свойств-enum использует встроенный fast-path и подставляет их числовое значение, **полностью игнорируя** зарегистрированный `SqlMapper.AddTypeHandler<T>` — даже после `SqlMapper.RemoveTypeMap(typeof(T))` (это НЕ помогает, `typeMap` и `typeHandlers` — разные внутренние словари Dapper). Подтверждено эмпирически через `general_log` MySQL: в ENUM(...)-колонку улетало `VALUES (0)` вместо `VALUES ('Fixed')`, что MySQL в strict-mode отбраковывает как "Data truncated". Чтение (десериализация ENUM-строки в C#-enum-свойство) через тот же хендлер работает корректно — баг только на запись.

**Фикс:** во всех репозиториях, где SQL пишет enum-поле (`InstrumentRepository`, `OperationRepository`, `MarketQuoteRepository`, `SignalRepository`, `OfferScheduleRepository`, `ProjectedCashFlowRepository`), параметр передаётся явным `.ToString()` в анонимном объекте, а не прямой передачей POCO целиком. `DapperTypeHandlers.cs` несёт подробный комментарий-предупреждение на этот счёт — любой новый Dapper-репозиторий с enum-полем должен повторять этот паттерн.

## Решение 2: G-спред — официальная методика MOEX (гауссовы поправки), а не интерполяция по узлам

**План указывал:** реконструкция кривой по параметрам Нельсона-Сигеля-Свенссона + поправки G1..G9.

**Решение:** первая реализация (этап 05) трактовала B1/B2/B3/G1..G9 как доли (0.10 и т.п.) и интерполировала G-поправки линейно между «стандартными узлами срока» — это было ошибочное предположение, не подтверждённое официальной методикой. При ревью (этапы 04-06) выяснилось: MOEX отдаёт эти параметры **в базисных пунктах** (реальный снимок: B1≈1521), а G-поправки — не узлы сплайна, а коэффициенты гауссовых «бугров» `exp(-(t-ai)²/bi²)` с ФИКСИРОВАННЫМИ центрами `ai`/`bi` (раздел 4.1 методики MOEX, утв. 07.07.2017, Протокол №40: `a1=0, a2=0.6, a(i+1)=ai+a2·k^(i-1), b1=a2, b(i+1)=bi·k, k=1.6`). Доходность получается через `Y(t)=exp(G(t)/10000)-1`. На реальном масштабе параметров старая реализация выдавала тысячи «процентов» доходности — юнит-тесты этого не поймали, т.к. использовали синтетические параметры в долевом масштабе (0.10), а не в б.п.

**Фикс:** `GSpreadCalculator`/`YieldCurveSnapshot` переписаны буквально по методике; добавлен регрессионный тест на реальной фикстуре `zcyc_gcurve.json`, проверяющий правдоподобный диапазон доходности (5–30%).

## Решение 3: Знак суммы операции для XIRR — источник истины один (брокер), не два

**План не указывал** конкретную реализацию знака.

**Решение:** первая реализация (`PortfolioXirrService`/`PortfolioSnapshotService`, этап 06) игнорировала фактический знак `Operation.AmountRub` через `Math.Abs()` и переписывала его по `OperationType` (Buy/Tax/Fee → минус и т.п.). При ревью выяснилось, что синк T-Invest (этап 04) уже копирует знак из брокерского поля `Payment` без модификации — иметь два независимых места, определяющих один и тот же знак, было риском расхождения, и реально ломало корректировки налога (`TaxCorrection*` мапится в тот же `OperationType.Tax`, но может быть положительным возвратом — старый код считал это оттоком).

**Фикс:** оба сервиса теперь используют `Operation.AmountRub` как есть, без перезаписи по типу. Единственный источник истины по знаку — брокер (через синк).

## Решение 4: `Instrument.DataIncomplete` обязателен к прокидыванию в проекцию потока

**План указывал** (§4.4): неполные данные помечаются, не подставляются молча.

**Решение:** первая реализация (этап 06) не прокидывала `Instrument.DataIncomplete` в `PositionCashFlowInput` вовсе — позиция с неполным графиком купонов от MOEX (пропущенный купон в «дырке» графика, который просто отсутствует в `Coupons`, не помечен `IsKnown=false`) выглядела в календаре полностью надёжной. Найдено при ревью этапов 04-06.

**Фикс:** добавлено поле `PositionCashFlowInput.DataIncomplete`, прокидывается из `CashFlowProjectionOrchestrator`; если `true` — все потоки результата помечаются `IsEstimated=true`.

## Решение 5: Контракт T-Invest по облигациям верифицирован отражением сборки, не по памяти

**План требовал** (§12.2) верифицировать контракт, не принимать на веру.

**Решение:** вместо документации/памяти — прямое отражение установленной сборки `Tinkoff.InvestApi 0.6.22.1`. Подтверждённый факт: `Bond` несёт только флаги `AmortizationFlag`/`FloatingCouponFlag` и одну дату `CallDate` (поля `PutDate` нет вообще) — полного графика амортизации/оферт T-Invest не отдаёт. Это буквально подтверждает преамбулу плана: MOEX остаётся единственным источником истины по расписаниям. Зафиксировано в `Connectors/TInvest/README.md`.

## Решение 8: Эвристика Put/Call по свободному тексту MOEX — консервативный дефолт на Put

**План не уточнял** алгоритм разбора `offertype`.

**Решение:** MOEX ISS возвращает тип оферты свободной русской строкой, без enum. Во всех собранных живых фикстурах встретилось ровно одно значение — «Оферта/Погашение» (put); ни одного реального call-примера не нашлось. Эвристика: всё, что не содержит "call"/"колл" → `Put`. Решение осознанно консервативное: put требует активного действия инвестора (spec §8 — легко пропустить, высокая ценность), false positive (call размечен как put) создаёт лишнее уведомление, false negative (пропущенный put) хуже. Задокументировано в `MoexBondizationParser` как ограничение данных, не догадка.

## Решение 9: Стакан (bid/ask) запрашивается у T-Invest, но не персистируется

**План** (§8) относит сигнал по низкой ликвидности к категории «на будущее».

## Решение 10: Ленивый резолв конфигурации в DI — обязательный паттерн

**План не уточнял** механику DI явно, только что секреты — из конфигурации.

**Решение:** все репозитории и `Options`-классы (например `JwtBearerOptions`) резолвят конфигурацию (connection string, секреты) **лениво из DI в момент создания**, а не в момент регистрации/в конструкторе синглтона — `services.AddScoped<IXxx>(sp => new Xxx(GetConnStr(sp)))`, и для опций — через `IConfigureOptions<T>`. Причина: `TestWebApplicationFactory.ConfigureWebHost` подменяет конфигурацию (`ConnectionStrings:DefaultConnection`, `Jwt:Secret` и т.п.) через `ConfigureAppConfiguration`, которая применяется ПОСЛЕ top-level кода в `Program.cs` — если читать конфиг раньше, интеграционные тесты будут видеть продовые/дефолтные значения, а не тестовый Testcontainer/секрет. Это правило обязательно для любого нового сервиса с конфигурацией.

## Решение 11: `FallbackPolicy.RequireAuthenticatedUser()` вместо точечного `.RequireAuthorization()`

**План указывал** защищать все доменные эндпоинты, явно не уточняя механизм.

**Решение:** глобальный `FallbackPolicy` в `Program.cs` требует аутентификацию по умолчанию на ВСЕХ эндпоинтах, кроме явно помеченных `.AllowAnonymous()` (`/health`, `/api/auth/telegram`). Причина: при добавлении новых доменных эндпоинтов на последующих этапах агент физически не может забыть защитить эндпоинт — он защищён по умолчанию. (Нюанс, найденный на этапе 08: Swagger middleware изначально был зарегистрирован ПОСЛЕ `UseAuthorization()` и сам попал под `FallbackPolicy`, отдавая 401 на `/swagger/v1/swagger.json` — исправлено перемещением блока Swagger до `UseAuthentication/UseAuthorization`.)

## Решение 12: Токен T-Invest через UI — `DataProtection`, не отдельный секрет шифрования

**План** (§7 этапа 00, §08) требовал шифрование, не уточняя механизм.

**Решение:** `Microsoft.AspNetCore.DataProtection` (`IDataProtectionProvider`) вместо ручного AES с отдельным секретом в конфиге — идиоматично для ASP.NET Core, не требует управления ещё одним ключом. Хранится в таблице `user_settings` (миграция `005_add_user_settings.sql`): `tinvest_token_encrypted` + `tinvest_token_last4` (открытым текстом, только для маски в UI). Источник токена — **только БД, без ENV-фолбэка** (явное решение владельца при подготовке к этапу 10: токен для T-Invest хранится исключительно на аккаунт, не как GitHub Secret/ENV-переменная деплоя).

`TInvestPortfolioClient` (этап 04) был переключён с DI-time `InvestApiClient` (токен из ENV `TInvest:Token` через `services.AddInvestApiClient(...)`) на ленивое построение клиента внутри scoped-экземпляра: `GetClientAsync(ct)` резолвит токен через `ITInvestTokenProvider.GetTokenAsync()` при первом обращении любого метода клиента, кэширует `InvestApiClient` на время жизни scope (один цикл синка), и бросает `InvalidOperationException` с понятным сообщением, если токен не задан — `BondSyncService`/`SyncCycleService` уже умеют деградировать на ошибке одного шага без падения всего цикла (spec §4.4). `services.AddInvestApiClient(...)` удалён из `DependencyInjection.cs`; `TINVEST_TOKEN` удалён из `backend.yml` (секретов GitHub Actions и ENV контейнера).

## Решение 14: `InvestedRub` = чистый капитал в позициях, не исторический объём покупок

**План** (этап 03) не давал формулу для `PortfolioValueSnapshot.InvestedRub`.

**Решение:** `Buy − Sell − Amortization − Redemption` (в знаковых величинах брокера, см. Решение 3) — то есть капитал, который физически сейчас «сидит» в открытых позициях, а не сумма всех покупок за всю историю. Может быть около нуля при полностью закрытом портфеле — это осознанно, не баг.
