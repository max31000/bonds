# Бэклог

Только actionable-задачи будущих циклов, кратко. История решений — git/`docs/history/`.
Формат: приоритет · суть · известные факты. Детальная проработка — в `plan/NN` при взятии в цикл.

## P1 — следующий бизнес-цикл

### 1. Флоатеры не должны попадать в топы доходности (подтверждённая дыра)
Пользователь не должен иметь возможность «случайно купить флоатер» через рекомендации.
Проверено по коду (2026-07): `BondUniverseEntry.IsFloater` (эвристика по BONDTYPE) есть, но
(а) `UniverseHygieneFilter` про флоатеры не знает → они видны в топ-10 сравнивалки и в скринере
при сортировке по YIELD, который для них несравним с YTM; (б) RV-медианы корзин и «дешёвые
соседи» флоатеров не исключают; (в) цель-флоатер можно выбрать в выпадашке и получить карточку
выгоды, сравнивающую несравнимое (позиции-флоатеры из сравнения исключены, цели — нет).
Решение обсудить: скрывать по умолчанию (новая причина в hygiene) vs жёсткая пометка + исключение
из сравнений/RV; в скринере — отдельный фильтр «только фикс-купон».

### 2. Переработка страницы «Рекомендации» → два блока (ТЗ владельца)
Сейчас 3+ блока, каждый решает задачу частично. Целевая структура:
- **Блок 1 «Слабые позиции → подобрать замену»**: список самых убыточных позиций; у каждой
  кнопка «подобрать замену» → выпадашка с режимами: «самые доходные рынка» / «самые надёжные» /
  «дешёвые соседи по корзине (RV)» / поиск. Один блок закрывает сценарий целиком — вместо
  нынешних «слабые звенья» + «матрица замен» + «RV-секции» по отдельности.
- **Блок 2 «Куда вложить сумму»**: конструктор остаётся, но кандидаты — весь рынок (сейчас
  пресет «максимум доходности» смотрит только на позиции пользователя — `GET /allocation`).
Убрать/слить лишние секции; общие принципы UX: меньше блоков, один сценарий = один блок.

## P2

### 3. Квазивалютные облигации — полноценная поддержка
Сейчас `IsOutOfScopeCurrency` выкидывает их из live-контура/метрик. Нужно: курс ЦБ/MOEX для
пересчёта номинала в ₽, метрики движка в валюте номинала + рублёвый эквивалент, отдельная
пометка в UI. По оценке — не сложно: FaceValue уже хранится в валюте, нужен курс и конверсия
на границах.

## Фичи из брейншторма 2026-07 (не взятые в первые волны)

- **C2. Пресеты-стратегии конструктора**: «сохранить дюрацию», «лесенка», «под ключевую ставку»
  (флоатеры — осознанно!), «как индекс RUCBITR».
- **C3. Лесенка / доход по месяцам**: цель «ежемесячный купон ~N ₽» → подбор бумаг с разнесёнными
  купонами (купонный календарь уже есть).
- **B5. Рыночный слой на scatter**: полупрозрачное облако банка под точками портфеля.
- **D1. Корзина сделок → кнопка «Купить»**: исполнение через T-Invest orders API. Требует
  ПОЛНОГО токена (текущий read-only!), слоя безопасности (подтверждения, лимиты) и закрытия
  P1-1/P1-2. Конструктор (задача 29) структурно уже готовит корзину.
- **D2. Алерты по банку**: сохранённый фильтр скринера + Telegram-сигнал при появлении бумаги.
- **D3. Календарь первичных размещений** (данные MOEX).
- **D4. Бенчмарк портфеля**: XIRR vs RGBITR/RUCBITR/депозитная ставка на дашборде.
- **A4. Рейтинги** (исследовательская): бесплатного API нет; жить листингом+G-спредом, либо
  ручной импорт.

## Техдолг (из ревью/аудитов, не блокирует)

- `GET /api/universe`: фильтрация/сортировка in-memory по ~3400 строк на каждый запрос + N+1
  по инструментам в `ResolvePortfolioIsinsAsync` — перенести в SQL при росте трафика/каталога.
- Кэш отсортированных спредов корзин в `RelativeValueService` (сейчас O(n²) на портфель — при
  текущих объёмах ок).
- N+1 по инструментам/расписаниям в `PortfolioHoldingsBuilder` (pre-existing, этап 08).
- Бэкфилл XIRR: амортизация учитывается текущим номиналом (дисклеймер есть; точный учёт — по
  графикам амортизаций).
- P-4 аудита: «купоны получено» и cost basis — до НДФЛ на купоны (в модели нет типа «налог на
  купон»); пересмотреть перед «действиями с портфелем».
- Лот = 1 бумага в аллокации/конструкторе (поля лота нет в модели; для корп. облигаций верно
  почти всегда).
- Фронт: чанк >500kB (code-splitting); `act()`-варнинги SegmentedControl; список секторов
  скринера захардкожен (зеркалит `BondUniverseSectorMapper`).
- Живой визуальный прогон 375px/тёмной темы (за Telegram-JWT — только на стенде владельца).
- Авто-оценка комиссии включает сервисные сборы (сырой тип T-Invest-операции не хранится) —
  при необходимости точности хранить сырой тип.
- `bond_universe`: нет soft-delete выведенных с торгов бумаг (отсеиваются фильтром косвенно);
  для длинной RV-истории может понадобиться явный признак.
