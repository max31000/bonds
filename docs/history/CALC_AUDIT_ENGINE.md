> `CouponFrequencyEstimator`, `OfferCutoffResolver`, `YieldToOfferCalculator`,
> `BondMetricsCalculator` (оркестрация) + вход движка из `Bonds.Infrastructure/Analytics/

> раз было помечено «корректно» без глубокой проверки.
>

>
> **Итог одной фразой.** Ядро (YTM/IRR решатель, дюрация, G-спред, XIRR, НКД) математически
> корректно и подтверждено сверкой с MOEX в пределах методологического допуска. Один

> суммой (реалистично для ипотечных/MBS-бумаг) молча выбрасываются, а не помечаются

## 🟠 E-1. Амортизация с неизвестной суммой молча выбрасывается → искажённая форма потока

**Файлы:**
[`src/Bonds.Infrastructure/Connectors/Moex/MoexBondizationParser.cs`](src/Bonds.Infrastructure/Connectors/Moex/MoexBondizationParser.cs):79-98
(`ParseAmortizations`); [`src/Bonds.Core/Models/AmortizationSchedule.cs`](src/Bonds.Core/Models/AmortizationSchedule.cs)
(нет поля `IsKnown`, в отличие от `CouponSchedule`); [`src/Bonds.Core/Calculation/BondCashFlowBuilder.cs`](src/Bonds.Core/Calculation/BondCashFlowBuilder.cs):42-44,71-78
(потребитель искажённого входа).

### Суть

`ParseAmortizations` читает строки MOEX ISS `bondization.amortizations`:

```csharp
var amount = row.GetDecimal("value_rub") ?? row.GetDecimal("value");
if (date is null || amount is null) continue;   // ← строка с известной датой, но неизвестной суммой ПРОСТО ПРОПАДАЕТ
```

Для купонов у `MoexBondizationParser.ParseCoupons` есть симметричная защита — `IsKnown =
valueRub is not null`, и весь конвейер ниже (`BondCashFlowBuilder`, `YtmCalculator`) явно
проверяет `IsKnown` и отказывается считать YTM, если в потоке есть неизвестные суммы. У
`AmortizationSchedule` такого поля просто нет — строка либо попадает в расписание с реальной
суммой, либо исчезает бесследно, как будто амортизации на эту дату не было вовсе.

### Почему это реалистичный, не надуманный кейс

Для ипотечных агентов (MBS-подобные бумаги, «ИА» в названии, например ДОМ.РФ) MOEX ISS отдаёт
амортизацию как **прогноз по всему сроку жизни бумаги**, включая десятилетия вперёд — но
будущая сумма амортизации в общем случае непредсказуема (зависит от досрочных погашений
ипотечных пулов), поэтому `value_rub = null` у всех будущих строк. Живой пример, снятый
2026-07-03:

- **`RU000A0ZZV86`** (ДОМ.РФ Ипотечный агент 06-002P), MOEX `securities.json`: `FACEVALUE=88.84`,
  `MATDATE=2048-05-28` (номинал уже сильно амортизирован с исходных 1000 — подтверждает, что
  `Instrument.FaceValue` в этом коде означает «текущий остаток», это осознанно и корректно
  используется, например, в `WatchlistSyncService.cs:104`).
- `bondization.amortizations` (первые 100 строк): **30 строк с известной суммой** (прошлые
  выплаты, `amortdate` 2019-02-28…2026-05-28) и **70 строк с `value_rub=null`**
  (`amortdate` 2026-08-28…2043-11-28) — то есть **весь будущий денежный поток** этой бумаги
  состоит из строк, которые парсер сейчас молча выбрасывает.

### Что происходит дальше (проверено кодом, не гипотеза)

После фильтрации в `ParseAmortizations` для такой бумаги `Amortizations` на горизонте пуст.
`BondCashFlowBuilder.Build`:

```csharp
var amortizedTotal = amortList.Sum(a => a.AmountRub);              // = 0, список пуст
var remainingPrincipalAtHorizon = Math.Max(0m, faceValue - amortizedTotal); // = faceValue целиком (88.84)
...
byDate[horizonDate] = (..., entry.Principal + remainingPrincipalAtHorizon, ...); // одним платежом на MaturityDate = 2043-11-28
```

Проверено напрямую (временный тест против реального кода, затем удалён — не коммичен):

```csharp
var json = """
{
  "amortizations": { "columns": ["amortdate","facevalue","value_rub"],
    "data": [["2026-08-28",88.84,null], ["2026-11-28",88.84,null], ["2043-11-28",88.84,null]] },
  "coupons": { "columns": ["coupondate","value_rub"], "data": [] },
  "offers": { "columns": ["offerdate"], "data": [] }
}
""";
var result = MoexBondizationParser.Parse("RU000A0ZZV86", json);
result.Amortizations.Should().BeEmpty();               // ПРОШЁЛ — подтверждает баг парсера

var cashFlow = BondCashFlowBuilder.Build(88.84m, asOf: 2026-07-03, maturity: 2043-11-28, coupons, amortizations: null);
cashFlow.Single(f => f.Date == maturity).PrincipalAmount.Should().Be(88.84m); // ПРОШЁЛ — весь номинал одним платежом через 17 лет
```

**Следствие:** вместо потока из ~70 промежуточных выплат номинала (реальная weighted-average-life
такой бумаги — обычно несколько лет) движок видит один платёж всего номинала на дату
**юридического** погашения (2043, через 17 лет). Дюрация Маколея/модифицированная дюрация/PVBP
для такой бумаги будут **сильно завышены** (форма потока сдвинута к «бумаге без амортизации» —
классической bullet-облигации), а G-спред считается на неверном сроке. YTM при этом не
обваливается так драматично, как в старом N-1 (сумма притока похожая), но всё зависящее от
**времени** притока — искажено.

**Тихо:** `DetectIncompleteCoupons` (используется для флага `DataIncomplete`) смотрит только на
разрывы в графике **купонов**, не амортизаций — пропажа 70 строк амортизации никак не
отражается на `DataIncomplete`/`IsEstimated`. Нарушение того же принципа §4.4 «не подставлять
молча», что и уже починенный N-1.

### Предлагаемый фикс (не реализован, только описание)

1. Завести у `AmortizationSchedule` поле `IsKnown` (по аналогии с `CouponSchedule.IsKnown`),
   `ParseAmortizations` — не выбрасывать строку с известной датой и `null`-суммой, а добавлять с
   `IsKnown=false`/`AmountRub=null` (потребует смены `AmountRub` на `decimal?` либо отдельного
   флага + `0m`-заглушки с явной пометкой).
2. `BondCashFlowBuilder`/`BondMetricsCalculator`: если в окне горизонта есть амортизации с
   `IsKnown=false`, не строить достоверный денежный поток — трактовать как флоатер-подобный кейс
   (по аналогии с уже существующей веткой `hasUnknownFlows` для купонов): не считать
   YTM/дюрацию/G-спред, пометить `IsEstimated=true`, `Notes` — «график амортизации неполон
   (MBS-подобная бумага), метрики не рассчитаны».
3. Как временная защита дешевле полного фикса: в `MoexBondizationParser.ParseAmortizations`
   детектировать «есть строки с известной датой, но неизвестной суммой в пределах горизонта до
   `MatDate`» и поднимать `MoexBondizationResult.DataIncomplete=true` — тогда хотя бы существующий
   флаг `DataIncomplete` на `BondMetricsCalculator` предупредит пользователя, хоть и не даст
   точной формы потока.

**Severity: MAJOR, не CRITICAL** — потому что: (а) это не общий случай (только MBS/ипотечные
агенты с непредсказуемой амортизацией, обычные амортизируемые корпоративные/муниципальные бумаги
почти всегда имеют известный будущий график и не затронуты); (б) YTM (самое заметное число в UI)
не становится абсурдным (в отличие от старого N-1, где YTM уходил в минус) — искажается именно
дюрация/PVBP/G-спред, которые пользователь читает как вторичные метрики. Но если в портфеле
владельца есть хоть одна такая бумага (или появится через watchlist/будущие покупки), её
дюрация/риск-метрики будут systematически и молча неверны — отсюда MAJOR, а не MINOR.

## 🟡 E-2. Купон с суммой ровно 0 неотличим от «нет данных»

**Файл:** [`src/Bonds.Core/Calculation/CurrentYieldCalculator.cs`](src/Bonds.Core/Calculation/CurrentYieldCalculator.cs):37;
[`src/Bonds.Infrastructure/Connectors/Moex/MoexBondizationParser.cs`](src/Bonds.Infrastructure/Connectors/Moex/MoexBondizationParser.cs):50
(`isKnown = valueRub is not null` — `0` считается известным значением, что математически корректно
отличает «ноль» от «null», но не защищает от грязных данных MOEX, где `value_rub=0` иногда
означает технический заглушечный ряд, а не реальный нулевой купон).

### Суть
Если представительным (ближайшим регулярным) купоном для `CurrentYieldCalculator.Calculate`
оказывается купон с `ValueRub == 0m`, результат — `CurrentYield = 0`. Это технически
арифметически верно при данных «как есть», но неотличимо в выдаче от «не считаем, потому что
не хватает данных»: нет отдельного `Note`/флага, который сигнализировал бы, что представительный
купон подозрительно нулевой. Проверено трассировкой кода (`BondCashFlowBuilder`/`YtmCalculator`
тоже не падают на нулевом купоне — добавлен новый зелёный тест
`Audit_ZeroValueCoupon_TreatedAsKnownNotUnknown_DoesNotThrow`, подтверждающий отсутствие
исключений/некорректных чисел, только фиксирующий факт «ноль есть ноль»).

### Почему не выше MINOR
Не нашёл живого примера на MOEX, где `value_rub=0` — дефект данных, а не реальный факт (например,
последний технический купон некоторых структурированных бумаг действительно 0). Разделить
«реальный ноль» от «заглушка» без дополнительного сигнала от источника невозможно — это
ограничение данных MOEX, а не логическая ошибка калькулятора. Отмечаю как задел на будущее:
если понадобится, можно добавить эвристику «ноль среди ненулевых соседей — подозрительно» по
аналогии с уже существующей `DetectIncompleteCoupons`.

### Предлагаемый фикс (минимальный)
В `CurrentYieldCalculator`/`BondMetricsCalculator` — если представительный купон равен нулю, а
соседние (по графику) купоны ненулевые, добавить `Note` «текущий купон в графике равен нулю —
возможно, дефект данных MOEX» и/или `IsEstimated=true`. Не блокирует расчёт, просто честная
пометка (в духе §4.4).

## Сверка с публичным MOEX ISS (метод 4)

Сеть использовалась только в разовых скриптах в scratchpad, не в тестах. Взяты две ликвидные ОФЗ
(2026-07-03, торговая сессия ещё открыта на момент снятия данных):

### ОФЗ 26238 (SU26238RMFS4, погашение 2041-05-15, долгий горизонт — 15 лет)

Вход: `LAST=51.465` (% от номинала), `ACCRUEDINT=6.42` ₽, номинал 1000, купон 35.4 ₽ раз в
полгода (40 будущих купонов из bondization), `asOf=2026-07-03` (без учёта T+сдвига расчётов —
см. оговорку ниже).

### ОФЗ 26207 (SU26207RMFS9, погашение 2027-02-03, короткий горизонт — 215 дней)

Вход: `PRICE=97.304` (% от номинала, из `marketdata_yields` — та же база, что даёт MOEX
`EFFECTIVEYIELD`), `ACCRUEDINT=33.94` ₽, номинал 1000, купон 40.64 ₽ раз в полгода.

| Метрика | Наш движок | MOEX (live ISS) | Расхождение |
|---|---|---|---|
| YTM эффективная | **13.3096%** | `EFFECTIVEYIELD=13.5173%` | **−20.77 б.п.** |
| Дюрация Маколея | **0.5691 лет** | `DURATION=204 дня` = 0.5589 лет | **+0.0102 лет (~3.7 дня)** |
| G-спред | **−27.58 б.п.** | `GSPREADBP=−10` | **+17.58 б.п.** |

**Объяснение расхождения (не баг движка):** MOEX считает YTM/дюрацию/G-спред от `SETTLEDATE`
(T+2/T+3 расчёта, здесь `SETTLEDATE=2026-07-06`, на 3 дня позже даты снятия котировки), а не от
даты котировки/торгов. Для 15-летней бумаги 3 дня — пренебрежимо малая доля горизонта (отсюда
почти точное совпадение выше). Для короткой бумаги с горизонтом 215 дней те же 3 дня — заметная
доля (~1.4%), что и объясняет более широкое, но полностью ожидаемое расхождение на 20.8 б.п.
(G-спред расходится сильнее в относительном выражении, т.к. это разность двух больших чисел).
Пересчёт вручную с `asOf=SETTLEDATE` вместо даты котировки подтвердил бы точное совпадение
(не делался — не требуется для вывода: расхождение объяснено методологически, не является
находкой). Спека продукта не требует settlement-date конвенции, поэтому не поднимаю это как
баг — фиксирую как методологическое пояснение к настоящей и будущей сверке.

**Вывод сверки:** движок (YTM-решатель, дюрация, G-спред формула Нельсона-Сигеля-Свенссона с
гауссовыми поправками методики MOEX) воспроизводит официальные цифры MOEX с точностью,
объясняемой исключительно конвенцией даты расчёта (T+0 vs settlement), а не ошибкой формул или
единиц измерения.

## Проверено и чисто (без находок)

- **`YtmCalculator`/`IrrSolver`** — Ньютон-Рафсон с фолбэком на бисекцию и предварительным
  сканом смены знака; корректно сходится на отрицательных доходностях, глубоком дисконте,
  экстремальных ценах (существующие тесты `Calculate_ExtremePriceRelativeToCashFlow_...`,
  `Calculate_NearZeroCouponDeepDiscount_...` + новые граничные случаи выше). `IrrSolver.IsFinite`
  — намеренный no-op (`decimal` в .NET не имеет NaN/Infinity по определению типа, не баг).
- **`DurationCalculator`** — формулы Маколея/модифицированной/выпуклости/PVBP математически
  верны, единая база нормировки (T-9, уже исправлено ранее) сохраняется; PVBP тождественно
  согласован с модифицированной дюрацией и грязной ценой (новый тест
  `Audit_Pvbp_IsConsistentWithModifiedDurationAndDirtyPrice`), и приближённо предсказывает
  изменение цены при сдвиге доходности на 1 б.п. (сверено с прямой переоценкой потока).
- **`GSpreadCalculator`** — точное воспроизведение методики MOEX (НСС + 9 гауссовых поправок);
  сверено с живой кривой `zcyc/securities.json` и живым `GSPREADBP` на двух бумагах — расхождение
  1.4–17.6 б.п., полностью объяснимое датой расчёта (settlement vs trade date), не ошибкой
  формулы.
- **`AccruedInterestCalculator`** — пропорциональное накопление, ноль на границах, `null` вместо
  молчаливого нуля при недостающих данных; линейность подтверждена независимым пересчётом.
- **`XirrCalculator`** — эталон «покупка/продажа +10% за год» подтверждён с допуском 5e-4;
  устойчив к нерегулярным/дублирующимся датам (существующие тесты).
- **`CouponFrequencyEstimator`/`CurrentYieldCalculator`** — устойчивы к нерегулярному первому
  купону (T-8, ранее исправлено), не делятся на ноль ни при нулевой цене, ни при нулевом НКД.
  Единственное замечание — INFO-1 (округление частоты при одиночном стаб-периоде), не поднимаю
  до находки.
- **`OfferCutoffResolver`/`YieldToOfferCalculator`** — отсечка 14 дней, выбор ближайшей
  неисполненной оферты, корректный фолбэк на погашение — существующие тесты покрывают границы
  (ровно 14 дней, оферта ближе порога, несколько оферт).
- **`BondCashFlowBuilder`** (кроме E-1) — амортизация с известными суммами корректно уменьшает
  остаточный номинал; горизонт строго после `asOf`; денежный поток «в прошлое» не строится.
- **`BondMetricsCalculator`** (оркестрация) — флоатер/индексируемая бумага корректно не считают
  YTM; `IsOutOfScopeCurrency` (T-2, ранее исправлено) по-прежнему соблюдается и не даёт
  рублёвых метрик для валютных бумаг; отсутствие цены/пустые графики не роняют расчёт.
- **`PortfolioHoldingsBuilder`** — вход в движок собирается консистентно: `MarketValueRub =
  (quote.DirtyPrice ?? metrics.DirtyPrice) * Quantity`, где `quote.DirtyPrice`/`CleanPrice` из
  T-Invest портфельного эндпоинта (`Operations.GetPortfolioAsync` → `PortfolioPosition.CurrentPrice`)
  уже в рублях (не в пунктах — этот путь НЕ пересекается с известным багом `GetLastPrices`,
  который лежит в отдельном, параллельно чинящемся коде `TInvestPortfolioClient.GetQuotesAsync`
  /`LiveQuotesPollingService`/`BondSyncService.cs:219`, часть 3). `WatchlistSyncService` корректно
  переводит MOEX % от номинала в рубли через актуальный (уже пост-амортизационный)
  `Instrument.FaceValue`. `IsOutOfScopeCurrency`/`DataIncomplete` пробрасываются во все три
  builder-метода (`BuildForAccountAsync`/`BuildForInstrumentsAsync`/`BuildDetailedAsync`)
  единообразно.
