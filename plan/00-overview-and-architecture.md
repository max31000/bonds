## 1. Что строим (одной фразой)

Персональный сервис **portfolio-aware bond analytics**: подтягивает позиции и операции с брокерского счёта Т-Инвестиций, обогащает их облигационной математикой (НКД, YTM, дюрации, выпуклость, PVBP, G-спред), строит персональный календарь денежного потока с НДФЛ, считает XIRR портфеля, генерирует сигналы (оферты, купоны, концентрация и т.д.) и визуализирует это. Один пользователь, один счёт, дневной батч-режим. См. §1–§3.

## 2. Жёсткие рамки MVP (не нарушать)

- **Только бесплатные источники**: T-Invest API + MOEX ISS. Ни один модуль не зависит от платного фида (§4, §11).
- **Вне скоупа** (закладывать точки расширения, но НЕ реализовывать): кредитные рейтинги, скринер по всей вселенной, мультисчёт/ИИС, полноценное налоговое моделирование, AI-слой, планирование вывода средств (§3 «Вне скоупа», §12.5).
- **Все расчёты — аналитические оценки, не инвестиционные рекомендации.** Дисклеймер обязателен в UI (§6, §11).
- **Брокерский токен — read-only, секрет.** Не логировать, не отдавать на фронт (§11).

### Разделение источников данных (принцип, действует во всех этапах)
- **T-Invest = актуальное «сейчас».** Текущая рыночная цена, стакан, и **НКД по конкретным открытым позициям** берутся из T-Invest (брокер отдаёт точные значения по счёту). К T-Invest ходим за оперативными данными при каждом синке.
- **MOEX ISS = справочное и историческое.** Графики купонов/амортизаций/оферт, параметры выпуска, кривая Gcurve, исторические ряды и ретроспектива. Меняется редко — кэшируем, ходим реже.
- При конфликте по **текущей цене/НКД открытой позиции** приоритет у T-Invest; по **расписаниям/справочнику** — у MOEX (§4.1/§4.2). Это осознанное уточнение «как» поверх §4.2 (спека оставляет выбор имплементатору). Движок (этап 05) умеет считать НКД сам — это fallback для бумаг без позиции (например, кандидатов на замену) и для сверки.

## 5. Карта инфраструктуры VDS (что уже есть и что добавляем)

VDS: IP `89.167.34.3`, домен `mvv42.ru`, SSH-пользователь `root`. nginx терминирует HTTPS (Let's Encrypt) и роутит по субпутям.

**Уже развёрнуто (не трогать):**
- `portal-shell` — главная `/var/www/portal/`, отдаётся на `location = /`.
- `cashpulse` — фронт `/var/www/cashpulse/`, бэк контейнер `cashpulse-api` на `127.0.0.1:5000`, nginx `location /api/` и `/health` → :5000, MySQL-контейнер `cashpulse-mysql` в сети `cashpulse-network`.
- `credit_calc` — проксируется на контейнер `:8081`.

**Добавляем для bonds:**
- Фронт: `/var/www/bonds/`, nginx `location /bonds/`.
- Бэк: контейнер `bonds-api` на `127.0.0.1:5001`, nginx `location /bonds/api/` → `:5001` и `location /bonds/health` → `:5001`.
- БД: **общий MySQL-сервер** (тот же контейнер, что у cashpulse) — новая база `bonds` + отдельный пользователь. `bonds-api` подключается к сети MySQL-контейнера.
- Реестр: запись `bonds` в `portal-shell/registry.json`.

> ⚠️ **Точные имена контейнера/сети/публикацию порта MySQL агент обязан сверить на живом сервере по SSH** (см. этап 01). Имена в этом плане — ожидаемые, но реальность сверяется фактом.

## 6. Изоляция от соседних сервисов (критично)

- Свой порт `5001` (5000 занят cashpulse).
- Свой namespace API `/bonds/api/` (НЕ общий `/api/`).
- Деплой-скрипт оперирует **только** контейнером `bonds-api` (`docker stop/rm/run bonds-api`), не трогает чужие контейнеры/сети.
- Своя база `bonds` и свой MySQL-пользователь с грантами только на неё.

- Namespace/сборки: `Bonds.Api`, `Bonds.Core`, `Bonds.Infrastructure`, `Bonds.Tests`, `Bonds.IntegrationTests`. Solution: `Bonds.sln`.

## 9. Определение готовности этапа (Definition of Done) — общее

Каждый этап считается закрытым, только когда агент **фактически проверил и предъявил доказательства**:
1. Все пункты «Критерии приёмки» этапа выполнены.
2. `dotnet build Bonds.sln -c Release` — без ошибок (после появления решения).
3. `dotnet test` — все тесты зелёные (юнит + интеграционные, релевантные этапу).
4. Frontend (после этапа 09): `yarn typecheck`, `yarn test:run`, `yarn build` — без ошибок.
5. Команды проверки из раздела «Проверка» этапа отработали с ожидаемым результатом (вывод приложить).
6. Никаких секретов в коде/логах/доках.

> **Принцип:** не заявлять «готово» без запуска проверочных команд и предъявления вывода. Если что-то не проверено — так и писать.

## 10. Стратегия наращивания (про §12.3 «Tier 0→1»)

«Tier 0→1» из §12.3 — это **абстрактная классификация фич по зависимости** (что является базой бизнес-логики, а что наращивается поверх), а **не** предписание порядка реализации. Базовые фичи (синк позиций, справочник MOEX, НКД/грязная цена, календарь, базовая таблица) — фундамент; YTM/дюрации/G-спред/сигналы — надстройка над ним.

Порядок реализации в этом плане — **послойный снизу вверх** (хранилище → коннекторы → движок → аналитика → сигналы → API → фронт): это даёт лучшую тестируемость каждого модуля и устойчивые границы. Зависимость «база → надстройка» соблюдена (надстроечные метрики опираются на уже готовый фундамент). Сквозной рабочий продукт собирается к этапу 09. Calculation Engine (этап 05) изолирован и максимально покрыт тестами (§12.4) — обязательные краевые кейсы: флоатер, амортизация, оферта, неполные купоны.

## 11. Точки расширения (заложить интерфейсы, не реализовывать)

- `IRatingProvider` — заглушка под рейтинги (вне MVP).
- Мультисчёт: `Position`/`Operation` несут `AccountId`, но логика работает с одним счётом.
- `ICandidateScreener` — интерфейс под скринер по вселенной (вне MVP).
- AI-слой — навешивается поверх посчитанных метрик через API; в БД/контрактах не закладывать обязательных зависимостей.
