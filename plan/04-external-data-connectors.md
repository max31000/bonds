# Этап 04 — Внешние коннекторы: MOEX ISS и T-Invest

> **Зависит от:** этап 03 (хранилище).
> **Опора по требованиям:** §4 (источники), §4.4 (риски полноты), §10 модули 1–2, §12.2 (верификация контракта T-Invest).

## Цель этапа

Наполнить хранилище реальными данными из **двух бесплатных источников**:
- **MOEX ISS** — справочно-аналитический слой (графики купонов/амортизаций/оферт, параметры выпуска, котировки, кривая Gcurve), маппинг ISIN→SECID.
- **T-Invest (gRPC SDK)** — «истина про счёт»: позиции и журнал операций.

После этапа: по команде синка позиции и операции попадают из T-Invest в БД, а по каждому инструменту подтягиваются расписания и параметры из MOEX. Источник истины по **расписаниям** — всегда MOEX, даже если T-Invest что-то дублирует (§4.1, §12.2).

> **Принцип устойчивости (§4.4):** пропуски не падают и не заполняются нулями молча — позиция/инструмент помечается «данные неполные». Это обязательное поведение, покрытое тестами.

## Часть A — Reference Data: коннектор MOEX ISS (`Bonds.Infrastructure/Connectors/Moex`)

Бесплатный, без регистрации, JSON. `HttpClient` через `IHttpClientFactory`. Эндпоинты (шаблоны из §4.2):

| Данные | Эндпоинт |
|---|---|
| Купоны + амортизации + оферты | `/iss/statistics/engines/stock/markets/bonds/bondization/{SECID}.json?iss.only=coupons,amortizations,offers` |
| Параметры выпуска + цены | `/iss/engines/stock/markets/bonds/securities/{SECID}.json` |
| Рыночные данные / стакан | marketdata / orderbook на соответствующем board |
| Кривая Gcurve (КБД) | `/iss/engines/stock/zcyc/securities.json` → параметры Нельсона-Сигеля-Свенссона (B1,B2,B3,T1,G1…G9) |
| Архив Gcurve | Gcurve на закрытие дня (для ретроспективы) |

> **Роль MOEX в этом проекте = справочно-историческая.** Из MOEX берём расписания (купоны/амортизации/оферты), параметры выпуска, Gcurve и исторические ряды. **Текущую рыночную цену, стакан и НКД по открытым позициям берём из T-Invest** (часть B), а не из MOEX marketdata/orderbook (см. «Разделение источников данных» в `00`). MOEX marketdata используем как fallback/для бумаг без позиции (кандидаты на замену) и для сверки. Это осознанное уточнение «как» поверх таблицы §4.2.

### Задачи
1. **Маппинг ISIN→SECID.** ISIN — общий ключ (§4.2). Реализовать резолвер ISIN→SECID (поиск по MOEX securities). Кэшировать результат в `instruments`.
2. **Парсеры** ответов в модели `CouponSchedule`/`AmortizationSchedule`/`OfferSchedule`/`Instrument`/`MarketQuote`. Форматы JSON ISS — массивы `columns`+`data`; писать устойчивые мапперы по именам колонок, не по индексам.
3. **Gcurve.** Распарсить параметры NSS в `YieldCurveSnapshot` (используются этапом 05 для G-спреда).
4. **Кэширование** справочных данных (меняются редко, §10 модуль 2): не дёргать MOEX на каждый расчёт; обновлять по расписанию/синку.
5. **Обработка неполноты (§4.4):**
   - Неполные купоны в `bondization` → пометить инструмент `data_complete=false`, сохранить что есть, не подставлять нули.
   - Флоатер: будущие купоны неизвестны — сохранять известные купоны до ближайшего пересчёта, `IsKnown=false` дальше.
   - Амортизация: сохранять график убывающего номинала.
   - Оферты: сохранять put/call с датами и `IsExecuted`.

### Тесты (юнит, `Bonds.Tests`)
- Парсеры на **сохранённых фикстурах** реальных ответов ISS (положить JSON-образцы в тест-ресурсы): обычная фикс-купонная бумага, амортизируемая, бумага с офертой, флоатер, **бумага с пропущенными купонами** (проверка флага неполноты).
- Парсер Gcurve → корректные параметры NSS.
- (Опц., вне CI) smoke-тест против живого MOEX по известному SECID, помеченный как `[Trait("Category","Live")]`, не блокирующий CI.

## Часть B — Broker Sync: коннектор T-Invest (`Bonds.Infrastructure/Connectors/TInvest`)

Официальный **gRPC SDK `Tinkoff.InvestApi`**. Токен **read-only**, из конфигурации `TInvest:Token` (секрет/ENV; либо введённый пользователем токен из БД — см. этап 08). Токен не логировать, не отдавать на фронт (§11).

### Задачи
1. **Клиент.** Инициализация `InvestApiClient` с токеном. Учитывать sandbox vs production (для разработки — sandbox/один счёт).
2. **Синк позиций.** Получить позиции портфеля с количеством и средневзвешенной ценой (cost basis) → upsert в `positions`. Маппинг инструмента: FIGI/ISIN из T-Invest → найти/создать `Instrument`, резолв SECID через MOEX (часть A).
3. **Синк журнала операций.** Покупки, продажи, купоны, амортизации, удержанный налог, комиссии → upsert в `operations` по `ExternalId` (идемпотентно). Это истина для XIRR (§4.1). **Удержанный налог в журнале — фактический** (T-Invest как налоговый агент); он идёт в XIRR. Не путать с проектируемым НДФЛ 13% на будущие купоны (этап 06) — это разные величины (см. также часть C / этап 06).
4. **Текущие рыночные данные.** По открытым позициям тянуть из T-Invest **текущую цену, стакан (стакан/ликвидность) и НКД** → писать в `market_quotes` с `Source=TInvest`. Это первичный источник «сейчас» для движка (этапы 05/06). MOEX-цена — только fallback.
5. **Верификация контракта (§12.2) — обязательно.** Точные имена сервисов/методов/полей по облигациям подтвердить по актуальному контракту SDK (Instruments/Operations/Portfolio/Coupons), а не принимать на веру. Зафиксировать в комментарии/доке, какие поля реально приходят (наличие амортизации/оферт — обычно только флаг; полные расписания берём из MOEX).
6. **Идемпотентность и инкрементальность.** Повторный синк не плодит дубли; по возможности — синк операций по курсору/с даты последней.
7. **Безопасность токена.** Чтение из секрета; маскирование в логах; запрет передачи на фронт.

### Тесты
- Юнит: маппинг ответов T-Invest (на зафиксированных фикстурах/моках) в `Position`/`Operation`; идемпотентность upsert по `ExternalId`.
- Интеграционные: синк в тестовую БД с замоканным клиентом T-Invest (без реальных вызовов в CI).
- (Опц., вне CI) live-smoke с реальным sandbox-токеном — помечен `Live`, не в CI. **Токен запрашивается у пользователя**, в код не попадает.

## Часть C — Оркестрация синка

- Сервис `BondSyncService` (`Bonds.Core`/`Infrastructure`): один вызов = (1) синк позиций+операций из T-Invest, (2) для каждого инструмента — догрузка/обновление расписаний и параметров из MOEX, (3) обновление котировок и Gcurve, (4) пометки неполноты.
- Идемпотентный, логируемый (без секретов), устойчивый к частичным сбоям источника (один упавший инструмент не валит весь синк — помечается неполным).
- Точка вызова: вручную (эндпоинт force-refresh, этап 08) и по расписанию (этап 07).

## Критерии приёмки
- [ ] Реализован резолвер ISIN→SECID; результат кэшируется в `instruments`.
- [ ] MOEX-парсеры корректно разбирают купоны/амортизации/оферты/параметры/Gcurve на фикстурах (включая флоатер и амортизацию).
- [ ] Бумага с пропущенными купонами помечается «данные неполные», нули не подставляются, синк не падает.
- [ ] T-Invest gRPC-клиент синкает позиции (с cost basis) и операции; контракт по облигациям верифицирован и задокументирован (§12.2).
- [ ] Текущая цена/стакан/НКД по открытым позициям пишутся из T-Invest (`Source=TInvest`); MOEX используется как справочно-историческое и fallback.
- [ ] Фактический удержанный налог из журнала T-Invest сохранён в `operations` (для XIRR) и не смешан с проектируемым НДФЛ.
- [ ] Повторный синк идемпотентен (нет дублей операций/позиций).
- [ ] Токен T-Invest читается из секрета, не логируется, не уходит на фронт.
- [ ] `BondSyncService` выполняет полный цикл и устойчив к частичным сбоям источника.
- [ ] Юнит- и интеграционные тесты коннекторов зелёные (live-тесты вне CI).
