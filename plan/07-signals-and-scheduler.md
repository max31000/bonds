# Этап 07 — Движок сигналов и планировщик

> **Зависит от:** этапы 04 (синк), 05 (метрики), 06 (аналитика/проекция).
> **Опора по требованиям:** §8 (триггеры/сигналы), §3 «Синхронизация 1–2 раза в первые часы торгов MOEX + форс-обновление», §11 (дневной батч).
> **Эталон фоновых задач:** cashpulse `Infrastructure/Services/ExchangeRateRefreshService.cs` — `IHostedService` (singleton) + `IServiceScopeFactory` для scoped-зависимостей.

## Цель этапа

Два связанных слоя:
1. **Signals Engine** (`Bonds.Core/Signals`) — правила-триггеры по §8, генерируют `Signal`.
2. **Scheduler** (`Bonds.Infrastructure/Scheduling`) — автосинк в окна первых часов торгов MOEX + обработка форс-обновления из UI; после синка пересчитывает метрики/проекцию и прогоняет сигналы.

После этапа сервис сам обновляет данные по расписанию и складывает сигналы в БД для показа на фронте.

## Часть A — Signals Engine (§8)

Каждый триггер — правило + настраиваемый параметр; на выходе `Signal` (тип, важность, привязка к позиции/инструменту, предлагаемое действие, дата). Реализовать как набор чистых правил (тестируемых) поверх данных позиций/расписаний/метрик:

| Триггер | Параметр | Важность |
|---|---|---|
| Приближается купон / амортизация / погашение | за N дней | обычная |
| **Приближается оферта (put/call)** | за N дней | **высокая (приоритетный)** — put нужно активно подавать через брокера, легко пропустить |
| Пересчёт купона у флоатера / смена ставки | — | обычная |
| Накопился незаинвестированный кэш (купоны) выше порога | порог | обычная |
| Доходность удерживаемой бумаги ниже сопоставимой по сроку альтернативы из портфеля > X б.п. | X б.п. | обычная |
| Нарушение лимита концентрации по эмитенту | порог | высокая |
| Дрейф дюрации портфеля от целевой | если задана `TargetAllocation` | обычная |
| Низкая ликвидность по позиции (тонкий стакан) | — | низкая (на будущее) |

- Рейтинговые сигналы — **вне MVP** (рейтинги за скоупом, §8, §3).
- Параметры триггеров — настраиваемые (хранить в настройках пользователя/`TargetAllocation`), с разумными дефолтами.
- Дедупликация: повторный прогон не плодит дубли одного и того же актуального сигнала (ключ: тип + позиция + дата/окно). `IsRead`/`IsDismissed` для UI.

### Тесты (`Bonds.Tests`)
- Каждый триггер: синтетический вход, при котором он срабатывает, и вход, при котором не срабатывает.
- Приоритет оферты: сигнал по приближающейся put-оферте имеет высокую важность.
- Концентрация по эмитенту: срабатывает при превышении порога.
- Дедупликация: повторный прогон не создаёт дублей.

## Часть B — Scheduler (§3, §11)

- `IHostedService` (singleton) + `IServiceScopeFactory` для доступа к scoped-сервисам (паттерн cashpulse).
- **Окна автосинка:** 1–2 запуска в первые часы торгов MOEX, по московскому времени (учесть таймзону сервера — приводить к Europe/Moscow явно). Конфигурируемое расписание. Realtime не нужен (§11).
- **Торговый календарь MOEX.** Пропускать выходные и биржевые праздники — в нерабочий день биржи синк бессмысленен (данные не обновляются, Gcurve не публикуется). Минимум — пропуск сб/вс; лучше — учёт праздников MOEX (можно определять по факту: если MOEX не отдаёт свежих торгов/Gcurve за день — считать день неторговым и не перетирать данные). Не падать в неторговый день.
- Полный цикл по тику: `BondSyncService` (этап 04) → пересчёт метрик/снимков (этап 05) → пересчёт проекции потока (этап 06) → **запись ежедневного снимка `portfolio_value_snapshots`** (NAV + XIRR на дату, для кривой §9) → прогон Signals Engine → запись сигналов.
- Устойчивость: ошибка источника не валит весь цикл; логирование без секретов; следующий тик повторит.
- **Force-refresh:** общий метод запуска того же цикла, вызываемый из эндпоинта (этап 08) по кнопке в UI. Защита от параллельного запуска (флаг «синк идёт»).

### Тесты
- Интеграционный: ручной запуск цикла на тестовой БД (с замоканными коннекторами) обновляет данные и генерирует сигналы.
- Force-refresh не запускает второй параллельный цикл, если один уже идёт.

## Критерии приёмки
- [ ] Реализованы все триггеры §8 (кроме рейтинговых — вне MVP); параметры настраиваемы, дефолты заданы.
- [ ] Сигнал по приближающейся оферте — высокой важности.
- [ ] Дедупликация сигналов работает (повторный прогон без дублей).
- [ ] Scheduler как `IHostedService` запускает полный цикл в окна торгов MOEX; таймзона Europe/Moscow учтена; выходные/праздники биржи пропускаются без падения.
- [ ] Полный цикл: синк → пересчёт метрик → проекция → **снимок NAV/XIRR в `portfolio_value_snapshots`** → сигналы; устойчив к частичным сбоям.
- [ ] Force-refresh запускает тот же цикл и защищён от параллельного запуска.
- [ ] Тесты триггеров и цикла зелёные.

## Definition of Done
Сервис автономно обновляется в окна торгов MOEX и по форс-обновлению, пересчитывает метрики/проекцию и генерирует приоритезированные сигналы (с акцентом на оферты) без дублей. Готово к выставлению через API и UI.
