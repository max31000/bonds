# Этап 08 — Контракты Backend API

> **Зависит от:** этапы 02 (auth), 04 (синк), 06 (аналитика/проекция), 07 (сигналы).
> **Опора по требованиям:** §10 модуль 9, §9 (что показываем), §6/§11 (дисклеймер).
> **Эталон:** cashpulse `Api/Endpoints/*` — minimal API, один файл на ресурс, DTO, Swagger.

## Цель этапа

Выставить наружу всё, что нужно фронту (этап 09), через minimal API. Маршруты внутри приложения — `/api/...` (nginx навешивает `/bonds`, см. этап 01). Все доменные эндпоинты — `.RequireAuthorization()` (этап 02), кроме `/health` и `/api/auth/*`.

> Контракты этого этапа — это интерфейс между бэком и фронтом. Зафиксировать DTO явно и задокументировать в Swagger; фронт (этап 09) опирается на эти контракты.

## Эндпоинты (предложение; имена/формы уточнить под фронт)

| Метод | Путь (внутр.) | Назначение | Источник |
|---|---|---|---|
| `GET` | `/api/positions` | таблица позиций с метриками (YTM/тек.доходность, дюрация, G-спред, дни до погашения/оферты, тип купона, флаги) | этап 05/06 |
| `GET` | `/api/positions/{id}` | детальная карточка позиции/инструмента (расписания, метрики, флаги неполноты) | этап 03–06 |
| `GET` | `/api/cashflow` | календарь поступлений: по месяцам и позициям, брутто/налог/нетто, даты освобождения тела | этап 06 |
| `GET` | `/api/analytics/xirr` | XIRR портфеля + ряд динамики стоимости/доходности во времени | этап 06 |
| `GET` | `/api/analytics/composition` | композиция по эмитенту/сектору/типу купона/корзинам дюрации | этап 06 |
| `GET` | `/api/analytics/scatter` | точки «дюрация×доходность» + параметры Gcurve для наложения кривой | этап 05/06 |
| `GET` | `/api/analytics/comparison` | сравнение/сортировка позиций (с дисклеймером) | этап 06 |
| `POST` | `/api/analytics/replacement` | анализ замены между текущими позициями (учёт комиссий) | этап 06 |
| `GET` | `/api/signals` | активные сигналы (тип/важность/действие/дата) | этап 07 |
| `POST` | `/api/signals/{id}/read` | пометить сигнал прочитанным/закрытым | этап 07 |
| `POST` | `/api/sync` | форс-обновление (запуск цикла этапа 07) | этап 07 |
| `GET` | `/api/sync/status` | статус последнего/текущего синка (идёт/завершён/ошибка, время) | этап 07 |
| `GET/PUT` | `/api/settings` | настройки: пороги триггеров, `TargetAllocation`, базовая валюта | этап 03/07 |
| `PUT` | `/api/settings/tinvest-token` | ввод/замена read-only токена T-Invest пользователем | см. ниже |

### Токен T-Invest через UI
- `PUT /api/settings/tinvest-token` принимает токен, **шифрует и сохраняет в БД** (или в защищённом конфиге), **никогда не отдаёт обратно** на фронт (только статус «задан/не задан» и маскированный хвост). Синк (этап 04) использует токен из секрета ENV **или** из этого хранилища — приоритет согласовать (рекомендация: ENV для CI/прод, UI-ввод как override для личного использования). Токен не логировать (§11).

### Общие требования к контрактам
- DTO отдельно от доменных моделей; денежные — числом с явной валютой RUB.
- В каждом ответе с метриками — флаги качества/применимости (`isFloater`, `isEstimated`, `dataIncomplete`, `calculatedToOffer`) из движка (этап 05).
- **Дисклеймер** (расчёты — аналитические оценки, не рекомендации) — отдавать в ответах сравнения/метрик или единым полем `disclaimer` (§6, §11).
- Ошибки — единый формат (порт `ErrorHandlingMiddleware` из cashpulse). Неполные данные — не 500, а 200 с пометкой.
- Swagger (Swashbuckle) описывает все эндпоинты и DTO.

## Тесты (`Bonds.IntegrationTests`)
- Порт паттерна cashpulse `Endpoints/*Tests.cs` + `TestWebApplicationFactory`.
- Каждый эндпоинт: без токена → `401`; с токеном владельца → `200` и валидный контракт.
- `POST /api/sync` запускает цикл (с замоканными коннекторами) и возвращает статус; `GET /api/sync/status` отражает состояние.
- `PUT /api/settings/tinvest-token`: после сохранения `GET /api/settings` не возвращает сам токен (только флаг/маску).
- Эндпоинт с неполными данными возвращает `200` + флаг, а не ошибку.
