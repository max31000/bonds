# Этап 09a — Фронтенд: каркас приложения и таблица позиций

> Часть 1 из 3 этапа 09 (декомпозиция — см. [`09-frontend.md`](09-frontend.md), раздел «Декомпозиция этапа на 3 подэтапа»). Это наименьший подэтап: цель — минимальный сквозной путь «вход → таблица позиций», на котором проверяется вся инфраструктура фронта (роутинг, стор, API-клиент, iframe-интеграция), прежде чем наращивать более тяжёлую графику в 09b.
>
> **Зависит от:** этап 08 (API контракты), этап 02 (auth-фронт — уже реализован: `Login.tsx`/`useAuthStore`/`ProtectedRoute`/`api/client.ts` существуют, не пересоздавай).
> **Опора по требованиям:** §9 (визуальный слой — конкретно таблица позиций), §6/§11 (дисклеймер).
> **Эталоны:** cashpulse `cashpulse-web/` (структура `api/`, `store/`, `components/`, `pages/`, `hooks/`, `utils/`, `theme.ts`), `portal-shell/docs/service-routing-integration.md` (basename, iframe, postMessage).

## Цель подэтапа

Заменить текущую заглушку (`ComingSoon`) на настоящий каркас приложения с реальной навигацией и ОДНИМ полноценным экраном — таблицей позиций с расчётными метриками. После этого подэтапа: владелец логинится через Telegram (уже работает с этапа 02) и видит таблицу своих облигационных позиций с YTM/дюрацией/G-спредом и визуальными пометками флоатеров/неполных данных. Это минимальный «вертикальный срез» фронта (см. `plan/00-overview-and-architecture.md` §10 «Tier 0→1» — синк→таблица — базовая фича).

Графика (календарь, scatter, композиция, кривая XIRR), сигналы, форс-обновление и настройки — это 09b/09c, **НЕ делать здесь**.

## Часть B — Экран: таблица позиций

Контракт API (этап 08, `GET /api/positions`):
```json
{
  "positions": [{
    "positionId": 0, "instrumentId": 0, "issuer": "", "sector": "",
    "quantity": 0, "marketValueRub": 0, "currencyRub": "RUB",
    "couponType": "Fixed", "maturityDate": "2030-01-01", "horizonDate": "2030-01-01",
    "calculatedToOffer": false, "ytmEffective": 0, "currentYield": 0,
    "modifiedDuration": 0, "gSpread": 0,
    "isFloater": false, "isIndexed": false, "isEstimated": false, "dataIncomplete": false
  }],
  "disclaimer": ""
}
```

Реализовать (`pages/Positions.tsx`, главный экран приложения после логина — маршрут `/`):
- Таблица (Mantine `Table`): эмитент, сектор, кол-во, рыночная стоимость, доходность, дюрация, G-спред, дни до погашения/оферты (посчитать из `horizonDate`/`maturityDate` на клиенте), тип купона.
  - **Доходность:** если `isFloater`/`isIndexed` — показывать `currentYield` с явной пометкой «текущая, не YTM» (поле `ytmEffective` для этих бумаг будет `null`); иначе — `ytmEffective`.
- Сортировка по доходности (клиентская, по колонке) — с видимым дисклеймером рядом с заголовком/сортировкой о том, что более низкая доходность ≠ «хуже» без учёта срока/риска (§9).
- Визуальные пометки (бейдж/иконка): `isFloater` («плавающая»), `isIndexed` («индексируемая»), `dataIncomplete` («данные неполные»), `calculatedToOffer` («к оферте», не к погашению).
- Пустой портфель / ошибка загрузки — не падать, показать понятное состояние (Mantine `Alert`/empty state).
- Дисклеймер-компонент из части A — виден на этом экране.

## Часть C — Тесты (Vitest + Testing Library + MSW)

- Рендер таблицы позиций с замоканным `GET /api/positions` (MSW): обычная бумага, флоатер (проверить, что показана `currentYield`, а не `ytmEffective`), бумага с `dataIncomplete=true` (проверить пометку), пустой список.
- 401 на `/api/positions` → logout/redirect на `/login` (переиспользует логику этапа 02, новый тест на конкретном экране).
- Юнит-тесты форматтеров `utils`, которые использует таблица позиций (деньги RUB, дни до даты) — остальные форматтеры (для календаря/аналитики) — в 09b.
- Существующие тесты этапов 01/02 (`App.test.tsx`, `Login.test.tsx`, `useAuthStore.test.ts`) — не должны сломаться.
