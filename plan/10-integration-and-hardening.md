# Этап 10 — Интеграция, сквозная проверка и харднинг

> **Зависит от:** все предыдущие этапы (01–09).
> **Опора по требованиям:** §11 (нефункциональные), §12 (указания имплементатору), §3 (границы MVP).

## Цель этапа

Свести всё воедино и доказать, что продукт работает как MVP end-to-end на проде: реальный синк → метрики → календарь → аналитика → сигналы → UI, при соблюдении нефункциональных требований (§11). Закрыть харднинг и документацию.

> Это финальный гейт. «Готово» = предъявлен сквозной прогон на живых (или sandbox) данных и зелёные проверки, а не утверждение.

---

## Часть A — Сквозной E2E на проде

С реальным read-only токеном T-Invest (запросить у пользователя) и живым MOEX:
1. Запустить синк (`POST /bonds/api/sync`) → позиции и операции подтянулись; по инструментам пришли расписания/параметры/Gcurve.
2. Проверить, что таблица позиций показывает осмысленные метрики; флоатеры — текущую доходность с пометкой; амортизируемые — корректный поток; бумаги с офертой — расчёт к оферте.
3. Календарь показывает будущие поступления брутто/налог/нетто; даты освобождения тела видны.
4. Scatter накладывает бумаги на кривую Gcurve; композиция суммируется в 100%; XIRR-кривая строится.
5. Сигналы сгенерированы (особенно приближающиеся оферты — приоритетно).
6. Проверить деградацию на неполных данных: бумага с пропущенными купонами помечена, ничего не упало.

## Часть B — Нефункциональный харднинг (§11)
- [ ] **Бесплатность зависимостей:** в обязательном контуре только T-Invest + MOEX; платных источников нет.
- [ ] **Безопасность токена:** read-only; не в логах; не на фронте; в БД — зашифрован. Проверить логи контейнера на отсутствие утечек токена/секретов.
- [ ] **Синхронизация:** автосинк срабатывает в окна торгов MOEX (проверить расписание/таймзону); форс-апдейт из UI работает; параллельный синк не запускается.
- [ ] **Устойчивость:** падение одного источника/инструмента не валит весь синк — деградация с пометками.
- [ ] **Валюта:** RUB как базовая везде.
- [ ] **Дисклеймер:** присутствует в UI на ключевых экранах.
- [ ] **Изоляция:** свой порт 5001, своя БД `bonds`/пользователь, контейнер `bonds-api`; `/cashpulse/`, `/credit_calc/`, `/` целы.

## Часть C — Проверка точек расширения (§11 этапа 00, §12.5)
- [ ] Заложены интерфейсы/заглушки под рейтинги, скринер по вселенной, мультисчёт, AI-слой — но они **не реализованы** и не являются обязательной зависимостью.
- [ ] Налоговое моделирование ограничено плоским НДФЛ 13% на купоны (как требует MVP).
- [ ] Не-RUB инструменты помечаются как вне MVP и исключаются из агрегатов (мультивалютность — будущее расширение).

## Часть C2 — Юридические пометки (§4.3)

- [ ] Данные ISS — собственность MOEX: личное использование ок; на случай будущего публичного релиза в `README`/decision-доке зафиксировать необходимость проверки Terms of Use перед перераспределением данных.
- [ ] Рейтинги отсутствуют в обязательном контуре (платная зона, вне MVP).

## Definition of Done
MVP по §1–§9 работает end-to-end на проде, соблюдает нефункциональные требования §11, изолирован от соседних сервисов, задокументирован, точки расширения заложены без реализации вне-MVP фич. Все сборки, тесты и прод-smoke зелёные с предъявленными доказательствами.
