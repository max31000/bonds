> **Кому адресовано.** Этот файл читает агент-имплементатор (**Sonnet**). Это самостоятельный этап поверх готовых этапов 00–10. Идти строго по порядку: сначала весь **Tier A** (это исправление багов, которые видит пользователь), затем **Tier B** (сырость), затем **Tier C** (новые фичи прогноза). Не перепрыгивать.

## 0. Контекст: что именно сломано и почему (прочитать до кода)

Эти факты установлены ревью кода. Не доверяй им слепо — **верифицируй** при работе, но они задают направление.

1. **У бумаг нет ни имени, ни эмитента, ни сектора.** `BondSyncService.EnrichFromMoexAsync` ([src/Bonds.Infrastructure/Sync/BondSyncService.cs](../src/Bonds.Infrastructure/Sync/BondSyncService.cs)) заполняет `FaceValue`, `MaturityDate`, `CouponType`, `HasAmortization`, `HasOffers`, но **никогда не пишет `Issuer`, `Sector` и тем более имя**. У модели `Instrument` ([src/Bonds.Core/Models/Instrument.cs](../src/Bonds.Core/Models/Instrument.cs)) вообще нет поля имени. Парсер `MoexSecuritiesParser` читает `SHORTNAME`/`SECNAME`, но они отбрасываются. Колонки `issuer`/`sector` в БД (миграция 003) существуют, но всегда `NULL`. → Таблица позиций показывает «— / —», композиция портфеля по эмитенту/сектору вся в «Не определено», тултип скаттера «Позиция #id», разбивка денежного потока «#id».

2. **Системный баг единиц «доли vs проценты».** Доходности (`YtmEffective`, `YtmSimple`, `CurrentYield`, `GSpread`, XIRR, значения безрисковой кривой) хранятся и отдаются как **доли** (0.12 = 12%). Но `formatPercent` ([bonds-web/src/utils/format.ts](../bonds-web/src/utils/format.ts)) делает `` `${value.toFixed(2)}%` `` — **без ×100**. → Везде, где показывается доходность (таблица позиций, скаттер, XIRR), число в 100 раз меньше правды: «0.12%» вместо «12.00%». При этом `sharePercent` композиции отдаётся уже в процентах (0–100) и форматируется отдельным `formatSharePercent` — поэтому простое «добавить ×100 в `formatPercent`» безопасно, но **обязательно проверить каждый вызов** (см. A2).

3. **Скаттер «дюрация × доходность» сжат в угол.** В [bonds-web/src/pages/Analytics.tsx](../bonds-web/src/pages/Analytics.tsx) ось X строится по точкам и по безрисковой кривой, а кривая отдаётся на сетке сроков **0.25…30 лет** (см. `GetScatter` в [src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs](../src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs)). Портфель из коротких бумаг (1–4 года) визуально вжимается в левые ~10% графика, хвост кривой до 30 лет растягивает ось. Плюс баг единиц из п.2 делает ось Y нечитаемой. Отсюда «кривой косой непонятный график».

4. **Денежный поток нельзя сопоставить с позициями.** На [bonds-web/src/pages/Cashflow.tsx](../bonds-web/src/pages/Cashflow.tsx) график — агрегат по месяцам (брутто/налог/нетто), а таблица «по позициям» — суммарно за весь горизонт, и позиции подписаны как `#{positionId}`. Связи «в этом месяце платят вот эти бумаги» в данных нет: `MonthlyCashFlowDto` не содержит разбивки по позициям. Пользователь физически не может проверить, какие бумаги дают показанную прибыль.

**Установленные значения масштаба пользователя (определяют приоритеты):** план = «починить + усилить прогноз»; приоритеты прогноза — (1) кто/сколько/когда платит, (2) сценарии ставок, (3) траектория портфеля. Прогноз по замене бумаг в этот этап не входит.

## Tier A — Исправления (обязательно, строго по порядку A1→A4)

### A1. Идентификация бумаги: имя, эмитент, сегмент

**Цель.** У каждой позиции есть человекочитаемое имя (например «ОФЗ 26244»), эмитент и грубый сегмент (госбумаги / корпоративные / муниципальные). Это устраняет «нет названий, бедная информация» и «эмитент/сектор не парсится».

**Бэкенд.**

1. **Модель.** В [src/Bonds.Core/Models/Instrument.cs](../src/Bonds.Core/Models/Instrument.cs) добавить свойство `public string? Name { get; set; }` (короткое торговое имя для отображения). Поля `Issuer` и `Sector` уже есть — используем их.

2. **Миграция.** Создать `src/Bonds.Infrastructure/Migrations/006_add_instrument_name.sql` по образцу существующих (миграции — embedded resources `Migrations/*.sql`, прогоняются `MigrationRunner`; формат — чистый MySQL/InnoDB utf8mb4). Содержимое: `ALTER TABLE instruments ADD COLUMN name VARCHAR(255) NULL AFTER figi;`. Обнови `InstrumentRepository` ([src/Bonds.Infrastructure/Repositories/InstrumentRepository.cs](../src/Bonds.Infrastructure/Repositories/InstrumentRepository.cs)): добавить `name` в SELECT-маппинг, в INSERT и в `ON DUPLICATE KEY UPDATE` рядом с `issuer`/`sector`.

3. **Имя из уже имеющихся данных.** `MoexSecuritiesParser` уже отдаёт `ShortName`/`SecName`. В `EnrichFromMoexAsync` после получения `info` присвоить:
   - `instrument.Name = info.ShortName ?? info.SecName ?? instrument.Isin;` (имя есть почти всегда — это не «неполные данные»).

4. **Эмитент и сегмент из MOEX description.** Движковый эндпоинт `/iss/engines/.../securities/{SECID}.json` **не** содержит эмитента надёжно. Эмитент и тип бумаги лежат в **глобальном** описании бумаги: `/iss/securities/{SECID}.json` — блок `description` (это таблица «свойство→значение» с колонками вида `name`, `title`, `value`).
   - Добавить в `IMoexIssClient` ([src/Bonds.Infrastructure/Connectors/Moex/IMoexIssClient.cs](../src/Bonds.Infrastructure/Connectors/Moex/IMoexIssClient.cs)) метод `Task<MoexSecurityDescription?> GetSecurityDescriptionAsync(string secid, CancellationToken ct)` и реализовать в `MoexIssClient` (по образцу других вызовов: тот же `HttpClient`, тот же базовый URL ISS, кэширование как у справочных данных — описание меняется редко).
   - Создать парсер `MoexSecurityDescriptionParser` и DTO `MoexSecurityDescription { string? Issuer; string? TypeName; }`. Блок `description` парсится тем же `IssTable.Parse(root, "description")`; затем пройтись по строкам, читая пары `name`→`value`.
   - **ОБЯЗАТЕЛЬНО сначала верифицировать точные ключи**: до написания кода выполни реальный запрос (эндпоинт бесплатный, без авторизации), например для ОФЗ `https://iss.moex.com/iss/securities/SU26244RMFS2.json?iss.only=description` и для корпоративной бумаги, и посмотри фактические значения колонки `name`. Ожидаемые ключи: эмитент — `EMITENT_TITLE` (проверь!); тип бумаги — `TYPE` или `TYPENAME` (значения вида `ofz_bond`, `corporate_bond`, `subfederal_bond`, `municipal_bond`, `exchange_bond`). Не хардкодь ключ, не убедившись.
   - **Сегмент (`Sector`).** ISS не отдаёт отраслевой сектор бесплатно. Поэтому `Sector` на этом этапе — это **грубый сегмент рынка**, выведенный из `TYPE`/`TYPENAME`: гособлигации (ОФЗ) → «Гособлигации»; subfederal/municipal → «Муниципальные»; corporate/exchange → «Корпоративные»; иначе → `null`. Маппинг положить в парсер/резолвер чистой функцией с тестами. Не выдумывать отраслевой сектор, которого в источнике нет (§4.4 «не подставлять молча»).

5. **Заполнение в `EnrichFromMoexAsync`.** После резолва `secid` дополнительно дёрнуть `GetSecurityDescriptionAsync(secid)`; присвоить `instrument.Issuer = desc?.Issuer ?? instrument.Issuer;` и `instrument.Sector = MapSegment(desc?.TypeName) ?? instrument.Sector;`. Если описание не пришло — не падать, оставить как есть (устойчивость к частичным сбоям уже заложена в обёртке try/catch синка).

6. **Проброс имени через слои.** Добавить `Name` в:
   - `PortfolioHolding` ([src/Bonds.Core/Analytics/PortfolioCompositionInput.cs](../src/Bonds.Core/Analytics/PortfolioCompositionInput.cs)) и заполнить его в `PortfolioHoldingsBuilder.ToHolding` (рядом с `Issuer = instrument.Issuer`).
   - DTO: `PositionRowDto` ([src/Bonds.Api/Endpoints/PositionsEndpoints.cs](../src/Bonds.Api/Endpoints/PositionsEndpoints.cs)), `ScatterPointDto` и `ComparisonRowDto` ([src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs](../src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs)), `PositionCashFlowDto` ([src/Bonds.Api/Endpoints/CashflowEndpoints.cs](../src/Bonds.Api/Endpoints/CashflowEndpoints.cs)) — добавить `Name` (и проследить, чтобы `Issuer`/`Sector` тоже отдавались там, где их ещё нет).

**Фронтенд.**

7. В [bonds-web/src/api/types.ts](../bonds-web/src/api/types.ts) добавить `name: string | null` в соответствующие типы (`PositionRow`, `ScatterPoint`, `ComparisonRow`, `PositionCashFlow`).
8. **Таблица позиций** ([bonds-web/src/pages/Positions.tsx](../bonds-web/src/pages/Positions.tsx)): добавить **первой** колонку «Бумага» со значением `row.name ?? row.issuer ?? '—'`. Колонки «Эмитент»/«Сектор» оставить. Если `name` отсутствует — fallback на ISIN (ISIN добавить в DTO, если его там нет — он всегда есть на инструменте).

### A2. Единый и корректный формат процентов

**Цель.** Доходности и XIRR показываются в правильном масштабе; G-спред — в базисных пунктах. Устранить системный баг ×100.

**Конвенция (зафиксировать и не нарушать):** на бэкенде все доходности — **доли** (0.12). На фронтенде форматтер сам умножает на 100.

1. В [bonds-web/src/utils/format.ts](../bonds-web/src/utils/format.ts) исправить `formatPercent`: `return \`${(value * 100).toFixed(2)}%\`;`.
2. **Проверить каждый вызов `formatPercent`** (их немного: Positions.tsx, Analytics.tsx scatter-тултип, XirrWidget). Все они передают **доли** — значит ×100 для всех корректно. `formatSharePercent` **не трогать** (он применяется к `sharePercent`, который уже в процентах 0–100).
3. **G-спред.** Сейчас в Positions.tsx он выводится через `formatNumber(row.gSpread)` → «0.02» для 200 б.п. Добавить в format.ts `formatBp(value)`: `` `${Math.round(value * 10000)} б.п.` `` (для `null` → «—»). Применить к G-спреду в таблице позиций и в любом сравнении. В заголовке колонки уточнить «G-спред, б.п.».
4. Тест-кейсы (vitest, рядом с существующими в `bonds-web/src/`): `formatPercent(0.1234) === '12.34%'`, `formatPercent(null) === '—'`, `formatBp(0.015) === '150 б.п.'`.

**Definition of Done A2.** Доходность ОФЗ в таблице выглядит как «~15.00%», XIRR — реалистичным процентом, G-спред — в б.п. Прогнать `npm test` в `bonds-web` — зелено.

### A3. Выпрямить скаттер «дюрация × доходность»

**Цель.** График читаемый: точки портфеля распределены по ширине, кривая не растягивает ось до 30 лет, ось Y — в процентах.

Файлы: [bonds-web/src/pages/Analytics.tsx](../bonds-web/src/pages/Analytics.tsx) (`ScatterWidget`) и при необходимости `GetScatter` в [src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs](../src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs).

1. **Ось X по данным.** Вычислить на фронте диапазон дюраций точек: `dMin = min(modifiedDuration)`, `dMax = max(modifiedDuration)`. Задать `XAxis` `domain={[Math.max(0, dMin - 0.5), dMax + 0.5]}` и `type="number"` `allowDataOverflow`. **Кривую обрезать** по этому же диапазону: рисовать только точки кривой с `termYears` внутри `[dMin - 0.5, dMax + 0.5]`, плюс по одной граничной точке слева/справа, чтобы линия доходила до краёв (можно запросить у бэка значение кривой ровно на `dMin`/`dMax`). Хвост до 30 лет на портфеле коротких бумаг не рисовать.
   - Допустимая альтернатива: добавить в `GetScatter` query-параметры границ и считать сетку кривой по фактическому диапазону дюраций портфеля (плотнее: шаг ~0.25–0.5 года в пределах диапазона). Если делаешь так — сетку строить динамически, не хардкодом `0.25…30`.
2. **Ось Y в процентах.** Точки и кривую для построения переводить в проценты (`effectiveYield * 100`, `curveYield * 100`) при формировании данных графика; `YAxis` оставить `unit="%"`, тултип — через обновлённый `formatPercent` (он теперь сам ×100, поэтому в тултип передавать **долю**, а в данные графика — процент; не запутаться: значение для оси Y — процент, значение для `formatPercent` в тултипе — исходная доля). Проще всего: в `point` держать оба — `yieldFraction` (для тултипа) и `yieldPercent` (для оси).
3. **Кривая — настоящая линия.** Точки кривой сортировать по `termYears` по возрастанию (recharts соединяет в порядке массива). Убедиться, что легенда отличает «Безрисковая кривая» (линия) от точек бумаг (маркеры).
4. **Пустое/вырожденное состояние.** Если у всех точек одинаковая дюрация (одна бумага) — не делать нулевой ширины домен: подмешать ±0.5 года (уже учтено формулой выше).

**Definition of Done A3.** На скриншоте видно: точки бумаг распределены по ширине графика, безрисковая кривая проходит рядом с ними (не «лежит на нуле»), ось Y подписана реалистичными процентами. Обновить/добавить тест `Analytics.test.tsx` на то, что домен X считается от данных, а не фиксирован.

## Tier B — Устранение сырости (после Tier A)

Мелкие, но заметные вещи. Каждая — отдельный маленький коммит.

1. **Заголовок «Дней до даты»** в таблице позиций неинформативен. Заменить на «Дней до погашения/оферты» и в ячейке, если `calculatedToOffer`, добавлять подпись «(оферта)».
2. **Пояснение «оценочных» потоков.** Бейдж «оценочно»/«есть оценочные потоки» нигде не объяснён. Добавить тултип/сноску: «Оценка: будущий купон флоатера неизвестен, посчитан по текущей ставке (§4.4, §6)».
3. **Пустые состояния с пользой.** Где «Нет данных» — добавить причину и следующий шаг (нет токена → ссылка в настройки уже есть в Positions; на Analytics/Cashflow — «появится после синхронизации»). Свериться, что пустые ответы при отсутствии счёта (уже реализованы в эндпоинтах) не показывают сырые ошибки.
4. **Идентификаторы в подписях.** Убрать из пользовательского UI «голые» `InstrumentId`/`PositionId` везде, где после A1 есть имя. ID оставить только как `data-testid`/ключи React.
5. **Дисклеймер про оферту в Positions** уже есть, но G-спред и дюрацию подписать единицами (б.п., лет) в заголовках колонок.
6. **Композиция «Не определено».** После A1 доля «Не определено» должна резко уменьшиться. Оставить её как честную категорию для бумаг без описания (делистинг/иностранные), но проверить, что она не доминирует на реальном портфеле.
7. **Формат денег/дат** — свериться, что `formatRub` и даты единообразны на всех экранах (они уже есть в utils — просто проверить применение).

**Definition of Done B.** Беглый проход по всем четырём экранам не оставляет «технических» артефактов (сырые id, необъяснённые бейджи, безразмерные числа). Существующие тесты зелёные.

## Tier C — Усиление прогноза (после Tier B)

Три возможности по приоритету пользователя. Все — поверх уже посчитанных движком метрик; **новых платных источников не вводить**, всё детерминированно и помечается дисклеймером «аналитическая оценка, не рекомендация» (§6, §11).

### C1. Доходный календарь: ясность «кто/сколько/когда» (приоритет 1)

В значительной части уже реализован в A4 (drill-down месяц→позиция). Здесь — довести до «инструмента»:

1. **Горизонт прогноза.** Добавить переключатель горизонта на экране денежного потока: 6 мес / 12 мес / 24 мес / весь срок. Параметр горизонта прокинуть в запрос `/api/cashflow` (если эндпоинт ещё не принимает `to`/`horizon` — добавить, проекция уже умеет ограничивать по дате).
2. **Сводка сверху.** Над графиком — карточки KPI: «Ожидается за 12 мес: нетто ₽X», «Купонами: ₽Y», «Возврат тела (амортизация+погашение): ₽Z», «Налог удержат: ₽W». Считать из тех же агрегатов.
3. **Ближайшие 5 поступлений.** Список «что прилетит первым»: дата, бумага, тип, нетто. Это самый частый вопрос пользователя.

**DoD C1.** На экране видно: сколько суммарно нетто за выбранный горизонт, чем именно (купон/тело), и поимённо ближайшие поступления. Тесты на расчёт KPI-сводки.

### C2. Сценарии ставок (приоритет 2)

**Идея.** У движка уже есть модифицированная дюрация и выпуклость по каждой бумаге — это ровно то, что нужно для оценки переоценки портфеля при сдвиге ставок. Формула второго порядка:

`ΔP/P ≈ −ModDuration · Δy + 0.5 · Convexity · Δy²`

где `Δy` — параллельный сдвиг доходности (в долях; например −0.01 = −100 б.п.).

**Бэкенд.**

1. Новый эндпоинт `GET /api/analytics/rate-scenario?shiftsBp=-200,-100,-50,0,50,100,200` (значения по умолчанию задать в коде). Для каждого сдвига и каждой бумаги (где есть `ModifiedDuration`; для флоатеров чувствительность к ставке мала — их `ModifiedDuration` ≈ до ближайшего пересчёта, использовать как есть, но пометить) посчитать `ΔP/P`, перевести в рубли через `MarketValueRub`, агрегировать по портфелю.
2. Чистый расчёт положить в `Bonds.Core` (например `RateScenarioService`) с юнит-тестами на формулу (известная дюрация/выпуклость → известный ΔP). Выпуклость уже считается движком (этап 05) — взять оттуда; если по бумаге её нет, использовать только первый член (−ModDur·Δy) и пометить как приближение.
3. Контракт ответа: `{ shiftsBp: int[], portfolio: [{ shiftBp, newValueRub, deltaRub, deltaPercent }], byPosition?: [...] }` + дисклеймер. Допущение «параллельный сдвиг всей кривой» зафиксировать в дисклеймере явно.

**Фронтенд** (на экране Аналитики, новый виджет `RateScenarioWidget`).

4. Слайдер/сегменты сдвига ставки (−200…+200 б.п.) и/или столбчатый график «стоимость портфеля при сдвиге»: ось X — сдвиг в б.п., ось Y — стоимость (или Δ%). Показывать текущую (Δ=0) как опорную. Под графиком — короткая фраза «При росте ставок на 100 б.п. портфель подешевеет примерно на ₽X (−Y%)».
5. Дисклеймер: оценка по дюрации/выпуклости, параллельный сдвиг, не учитывает кредитный спред и реинвест.

**DoD C2.** Виджет показывает осмысленную чувствительность: длинные бумаги дают больший Δ, короткие — меньший; при Δ=0 стоимость равна текущей. Тесты на `RateScenarioService` (включая ветку без выпуклости).

### C3. Траектория портфеля (приоритет 3)

**Идея.** Спроецировать стоимость/доход портфеля во времени с реинвестированием купонов — «как будет расти портфель, если ничего не менять».

**Бэкенд.**

1. Новый эндпоинт `GET /api/analytics/trajectory?horizonMonths=36&reinvest=true&reinvestRate=` . Алгоритм (детерминированный, помесячно):
   - Стартовая стоимость = текущая рыночная стоимость портфеля.
   - Каждый месяц прибавлять нетто-поступления из проекции денежного потока (купоны после НДФЛ + амортизация + погашение) — это уже считается.
   - Если `reinvest=true`: накопленный кэш реинвестируется по ставке `reinvestRate` (по умолчанию — текущая средневзвешенная доходность портфеля к погашению; вынести в параметр). Если `reinvest=false` — кэш просто копится отдельной линией «свободный кэш».
   - Выдавать две линии: «стоимость бумаг + реинвест» и «накопленный доход нарастающим итогом».
2. Чистый расчёт — в `Bonds.Core` (`PortfolioTrajectoryService`) с тестами (детерминированный вход → известный ряд). Явно пометить как оценку при допущении удержания бумаг до погашения и заданной ставки реинвеста.
3. Контракт: `{ points: [{ date, portfolioValueRub, cumulativeIncomeRub, reinvestedCashRub }], assumptions: {...}, disclaimer }`.

**Фронтенд** (виджет `TrajectoryWidget` на Аналитике): линейный график во времени (стоимость + накопленный доход), переключатель «реинвестировать купоны: да/нет», поле ставки реинвеста (по умолчанию — доходность портфеля). Подпись допущений.

**DoD C3.** График показывает растущую траекторию; включение реинвеста даёт более крутую кривую, чем без него; накопленный доход монотонно растёт. Тесты на `PortfolioTrajectoryService`.

## Стратегия тестирования (для всего этапа)

- **Бэкенд (`tests/Bonds.Tests`)**: чистые сервисы (`RateScenarioService`, `PortfolioTrajectoryService`, маппинг сегмента, парсер описания MOEX) — детерминированные юнит-тесты с фикстурами. Парсеры MOEX тестировать на сохранённых JSON-фикстурах (положить реальные ответы в `tests/Bonds.Tests/Fixtures/Moex/`, как уже сделано для `zcyc_gcurve.json`/bondization). **Не ходить в сеть из тестов.**
- **Интеграционные (`tests/Bonds.IntegrationTests`)**: новые/изменённые эндпоинты отдают корректную форму ответа и пустые ответы при отсутствии счёта.
- **Фронтенд (`bonds-web`, vitest)**: форматтеры (A2), drill-down денежного потока (A4), домен оси скаттера (A3), новые виджеты C2/C3 рендерятся с моками стора.
- Прогонять `dotnet test` и `npm test` (в `bonds-web`) после каждого Tier; **не** заявлять готовность без зелёных тестов и без визуальной проверки экранов (см. правило verification-before-completion).

## Definition of Done всего этапа

1. Реальный портфель в проде показывает бумаги с именами, эмитентами и сегментами; композиция и сигналы используют имена.
2. Все доходности/XIRR — в верном масштабе; G-спред — в б.п.
3. Скаттер читаем; денежный поток кликабелен до конкретных бумаг.
4. Работают три прогнозных блока: доходный календарь с горизонтом и KPI, сценарии ставок, траектория портфеля.
5. Все тесты зелёные; README/статус-док обновлён ссылкой на этот этап.

## Чего НЕ делать (рамки)

- Не вводить платные источники (рейтинги, отраслевые секторы из платных фидов, Cbonds) — §3 «Вне скоупа».
- Не строить скринер по всей вселенной бумаг и не добавлять мультисчёт — вне MVP.
- Сценарии ставок и траектория — **детерминированные оценки по уже посчитанным метрикам**, без стохастики/Монте-Карло и без ML. Это инструмент прогноза в смысле «проекция при заданных допущениях», а не вероятностная модель.
- Не менять модель безопасности токена и не выводить токен на фронт.
