## Проблема

Сервис считает метрики, но не отвечает на главные вопросы пользователя: «что из моего портфеля
слабое и на что его поменять» и «я закинул 15 000 ₽ — что докупить». При этом бэкенд-математика
замены **уже написана и работает**, но фронт её не вызывает вообще (мёртвая фича):

- `GET /api/analytics/comparison` — сравнительная таблица всех позиций (эффективная доходность,
  дюрация, G-спред, горизонт) — [AnalyticsEndpoints.cs](../src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs);
- `POST /api/analytics/replacement` — расчёт «держать A vs переложиться в B» с комиссиями обеих
  сделок — [SwitchAnalysisService.cs](../src/Bonds.Core/Analytics/SwitchAnalysisService.cs),
  [PositionComparisonService.cs](../src/Bonds.Core/Analytics/PositionComparisonService.cs).

## Что сделать

### A. Страница «Рекомендации» (frontend, ядро задачи)

Новая страница `bonds-web/src/pages/Recommendations.tsx` + роут + пункт навигации в
[AppLayout.tsx](../bonds-web/src/components/AppLayout.tsx) (между «Аналитика» и «Сигналы»).
API-клиент — новый `bonds-web/src/api/recommendations.ts`, store — `useRecommendationsStore.ts`
(по образцу существующих сторов). Три секции сверху вниз:

**1. «Слабые звенья» (sell-кандидаты).** По данным `comparison`: отранжировать позиции по
эффективной доходности с поправками — показать топ-3 с причинами-бейджами: «доходность ниже
медианы портфеля на X п.п.», «оферта через N дней», «доля в портфеле Y%» (концентрацию считать на
фронте из `/api/analytics/composition` byIssuer). Флоатеры/индексируемые и `dataIncomplete` — не
ранжировать, показывать отдельной строкой «вне сравнения» (их текущая доходность несравнима с YTM —
см. дисклеймер `PositionComparisonService.YieldDisclaimer`).

**2. «Замены» (switch-матрица).** Для каждого sell-кандидата — топ-2 замены из остальных позиций:
дёрнуть `POST /api/analytics/replacement` для пар (кандидат × лучшие по доходности с сопоставимой
дюрацией ±1.5 года), показать карточками: «A → B: выгода ≈ X ₽ за горизонт Y лет, комиссии учтены,
окупится за Z мес». Отрицательная выгода → не показывать пару. Горизонт — из `daysToHorizon`
меньшей из двух бумаг. Не более ~6 запросов replacement за загрузку.

**3. «Куда вложить сумму».** Форма: `NumberInput` суммы (дефолт 15 000 ₽) → `GET
/api/analytics/allocation?amountRub=15000` (новый эндпоинт, см. B) → список: «Бумага X — купить
N шт по ~P ₽ (с НКД), потратите S ₽, эффективная доходность Y%», остаток «на счету останется R ₽».

### B. Эндпоинт аллокации (backend)

1. Чистый сервис `Bonds.Core/Analytics/CashAllocationService.cs` + юнит-тесты. Вход: сумма;
   кандидаты (текущие позиции) с полями: эффективная доходность, цена покупки одной бумаги
   (грязная: цена + НКД + комиссия по ставке `SwitchAnalysisService.DefaultCommissionRate`,
   переиспользуй константу), лот (у большинства корп. облигаций лот = 1 бумага — брать
   из инструмента, поле лота есть в данных T-Invest/Instrument; проверь
   [Instrument.cs](../src/Bonds.Core/Models/Instrument.cs), при отсутствии — считать 1 и пометить);
   текущая доля эмитента в портфеле. Алгоритм — жадный с ограничением концентрации:
   сортировка по эффективной доходности убыв., покупать максимум штук, пока доля эмитента
   после покупки ≤ `MaxIssuerSharePercent` (дефолт 25%, в options), переходить к следующей;
   остаток < цены самой дешёвой бумаги → стоп. Выход: список `(instrumentId, quantity,
   estimatedCostRub, effectiveYield)` + `leftoverRub` + флаги пропуска («пропущен: лимит
   концентрации», «пропущен: нет доходности»).
   Тесты: сумма меньше самой дешёвой бумаги → пустой список и весь остаток; лимит концентрации
   останавливает докупку лидера и пускает деньги во вторую бумагу; floater без YTM использует
   текущую доходность (та же логика `effectiveYield`, что в
   [PositionsEndpoints/PositionComparisonService]).
2. Эндпоинт `GET /api/analytics/allocation?amountRub=` в
   [AnalyticsEndpoints.cs](../src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs): собрать кандидатов
   через `PortfolioHoldingsBuilder` (как делают соседние эндпоинты), вызвать сервис, вернуть DTO
   с дисклеймером: «Оценка распределения свободных средств по бумагам текущего портфеля. Не
   учитывает налоги и не является инвестиционной рекомендацией». Валидация: `amountRub <= 0` → 422.
3. Интеграционный тест эндпоинта на засеянном портфеле.

### C. Тесты фронта

Vitest + MSW ([msw-handlers.ts](../bonds-web/src/test/msw-handlers.ts)): рендер трёх секций;
sell-кандидат с причинами; форма аллокации показывает результат и остаток; floater в «вне сравнения»;
дисклеймер присутствует.
