> доли на бэкенде, рубли decimal. Дисклеймеры обязательны. Коммиты с префиксом T-29.
>
> **Зависимости (все влиты):** 26 (банк + `GET /api/universe`), 27 (материализация
> `POST /api/universe/{secid}/materialize` + серверный поиск-компонент), 22 (комиссия из
> ICommissionRateProvider), 24 (разбивка НКД), 25 (налог — покупок не касается).
>
> **Прицел на будущее:** результат конструктора — это будущая «корзина сделок» под кнопку
> «Купить» (следующие итерации). Структурируй выход так, чтобы корзину можно было сохранить/
> исполнить, но САМУ кнопку/сохранение НЕ делай (YAGNI).

## Проблема

«Куда вложить сумму» — жадный алгоритм без участия пользователя: нельзя сказать «40% в эту,
30% в ту, остальное поровну». Нужен конструктор: сумма + корзина бумаг с процентами → штуки с
учётом лотов/НКД/комиссии + метрики корзины + **what-if**: как изменится весь портфель (доходность,
дюрация, концентрация) после покупки. What-if — недостающее звено перед кнопкой «Купить».

## Что сделать

### A. Расчёт корзины (Core, чистый сервис)

`Bonds.Core/Analytics/BasketBuilderService.cs` + юнит-тесты:
- Вход: `amountRub`, строки `{ instrumentId, targetWeightFraction }` (Σ весов ≤ 1; недобор —
  остаток деньгами), на каждую бумагу — цена лота (грязная + комиссия, как в CashAllocationService),
  эффективная доходность, дюрация, эмитент.
- Алгоритм: целевые рубли на бумагу = amount × вес → штук = floor(целевые/цена лота); недоиспользованные
  рубли каждой строки и нераспределённый вес — в `leftoverRub`. НЕ перераспределяй остатки жадно
  между бумагами без запроса — предсказуемость важнее оптимальности (задокументируй).
- Выход: строки (штук, стоимость с разбивкой clean/НКД/комиссия, фактический вес после округления),
  `leftoverRub`, метрики корзины: средневзвешенная доходность и дюрация (веса — стоимость;
  floater'ы в доходность не включать — сноска, тот же принцип, что «Итого» задачи 21).
- Тесты: веса 40/30/30 на 15к; вес не кратен лоту → остаток; Σ весов < 1; бумага дороже своей доли
  → 0 штук и остаток; floater вне средневзвешенной доходности.

### B. What-if превью портфеля (Core + endpoint)

1. `Bonds.Core/Analytics/PortfolioWhatIfService.cs`: вход — текущие holdings (стоимость, доходность,
   дюрация, эмитент по каждой позиции) + строки корзины из A. Выход — до/после: суммарная стоимость,
   средневзвеш. доходность, средневзвеш. дюрация, топ-концентрации по эмитентам (доля до → после),
   `warnings`: превышение лимита концентрации (порог — `DefaultMaxConcentrationPercent` из
   настроек, дефолт как в аллокации), появление эмитента > 25%. Юнит-тесты: дельты руками на
   3 позициях + корзине из 2 бумаг; предупреждение о концентрации срабатывает/не срабатывает.
2. `POST /api/analytics/basket` ([AnalyticsEndpoints.cs](../src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs)):
   вход `{ amountRub, lines: [{instrumentId, weightFraction}] }` → собрать входы (holdings через
   `PortfolioHoldingsBuilder`, для бумаг вне портфеля — `BuildForInstrumentsAsync`; ставка комиссии
   из `ICommissionRateProvider`), вызвать A и B, вернуть `{ basket, whatIf, disclaimer }`.
   Валидация: amountRub > 0, веса ∈ (0,1], Σ ≤ 1.0001, instrumentId существуют — иначе 422.
   `GET /api/analytics/allocation` НЕ трогать (обратная совместимость; жадный алгоритм остаётся
   как пресет, см. C.3).
3. Интеграционные тесты: корзина из позиции + watchlist-бумаги считается; 422-валидации;
   what-if дельты присутствуют.

### C. Frontend — «Куда вложить» становится конструктором

Секция AllocationSection в [Recommendations.tsx](../bonds-web/src/pages/Recommendations.tsx) —
вынеси в отдельный файл `bonds-web/src/components/BasketConstructor.tsx` (страница разрослась):

1. Сумма (как сейчас) + **строки корзины**: поиск бумаги (переиспользуй компонент серверного
   поиска по банку из задачи 27; источники — позиции, watchlist, весь банк; бумага из банка при
   добавлении материализуется через `POST /api/universe/{secid}/materialize`) + поле процента
   (NumberInput 1–100) + удалить строку. Индикатор Σ процентов (подсветка если ≠ 100).
2. Кнопка «Нормализовать до 100%» (пропорционально) и кнопка-пресет **«Максимум доходности»** —
   заполняет корзину результатом существующего `GET /api/analytics/allocation` (жадный алгоритм
   как отправная точка, дальше пользователь крутит проценты).
3. «Рассчитать» → `POST /api/analytics/basket` → результат:
   - таблица строк: бумага · штук · стоимость (с раскрытием clean+НКД+комиссия — паттерн задачи 24)
     · факт. вес vs целевой; остаток деньгами;
   - **метрики корзины**: доходность, дюрация (со сноской про floater'ы);
   - **блок What-if**: «Портфель после покупки»: стоимость ↑, доходность 14.4% → 15.1% (стрелка,
     цвет), дюрация 1.2 → 1.6, концентрации с ⚠️ на превышениях. Компактно, 3-5 строк.
4. Старый простой вывод жадной аллокации удалить (его заменяет пресет + конструктор); тесты
   секции переписать. Остальные секции страницы не трогать.
5. Vitest + MSW: добавление строки из поиска, нормализация, пресет заполняет корзину, расчёт
   рендерит штуки/метрики/what-if, ⚠️ концентрации, 422 → ошибка, Σ≠100 подсвечен.

## Критерии приёмки

- [ ] Корзина: проценты → штуки с лотами/НКД/комиссией; остатки предсказуемы; метрики корзины со сноской.
- [ ] What-if: до/после по доходности/дюрации/концентрациям с предупреждениями.
- [ ] Пресет «Максимум доходности» = старый жадный алгоритм; `GET /allocation` не сломан.
- [ ] Поиск бумаг — портфель + watchlist + банк (с материализацией).
- [ ] `dotnet build/test`, `yarn typecheck/test:run/build/lint` — зелёные.
