# Задача 30 — Relative Value: «дорогая/дешёвая» относительно своей корзины (B3) — ВОЛНА 2

> **Кому адресовано.** Агент-исполнитель. Задача самодостаточна. Конвенции — [CLAUDE.md](../CLAUDE.md):
> доли на бэкенде, б.п. делает фронт-форматтер. Коммиты с префиксом T-30.
>
> **Зависимости:** волна 1 (задачи 26–29) ВЛИТА В MAIN И ЗАДЕПЛОЕНА ЗАРАНЕЕ — банк уже копит
> `bond_universe_history` (дневные срезы yield/duration/gspread_approx). Используй накопленное;
> если истории < 5 дней — работай по последнему снимку с пометкой (см. B.3).
>
> **Идея (Bloomberg RV-подход).** «Где YTM больше» — плохой вопрос (наверху всегда преддефолтный
> мусор). Правильный вопрос деск-трейдера: «что дёшево ОТНОСИТЕЛЬНО СВОИХ» — сравнить спред бумаги
> с медианным спредом её корзины (сектор × дюрация). Бумага сильно НАД медианой корзины — «дешёвая»
> (рынок недооценивает или закладывает риск), сильно ПОД — «дорогая» (кандидат на продажу).

---

## Что сделать

### A. Корзины и статистика (Core, чистый сервис)

`Bonds.Core/Analytics/RelativeValueService.cs` + юнит-тесты:
1. **Корзина** = сектор × дюрационная корзина. Дюрационные границы возьми ТЕ ЖЕ, что использует
   `PortfolioCompositionService.byDurationBucket` (посмотри код — консистентность с существующим
   UI важнее красоты границ). Сектора — как в `bond_universe.sector`.
2. Вход: строки вселенной (secid, sector, durationYears, gspreadApproxFraction) — только НЕ
   скрытые гигиеническим фильтром (мусор в медиану не пускать!). Выход по каждой корзине:
   `median`, `p25`, `p75`, `count`.
3. **Минимальный размер корзины ≥ 5** бумаг; меньше → fallback на корзину «сектор целиком»,
   если и там < 5 → «весь рынок» — с понижением `confidence` (enum High/Medium/Low в результате).
4. **RV-оценка бумаги**: `deviationFraction = gspread бумаги − median корзины` (положительная =
   «дешёвая», спред выше медианы) + перцентиль внутри корзины + confidence. Для позиций портфеля
   спред бери НАШ ТОЧНЫЙ (`PortfolioHolding.GSpread`), для рыночных кандидатов —
   `gspread_approx`. Смешение точного и приближённого допустимо и ДОЛЖНО быть задокументировано
   doc-comment'ом (несоответствие конвенций даёт погрешность единицы б.п. — меньше типового
   отклонения).
5. Юнит-тесты: медиана/перцентили корзины руками; fallback-цепочка корзина→сектор→рынок;
   deviation знак/перцентиль; скрытые бумаги не попадают в статистику.

### B. Сглаживание по истории (Infrastructure)

1. Оркестратор `RelativeValueSnapshotBuilder` (Infrastructure): статистику корзин считать не по
   одному дню, а по **медиане дневных медиан за последние 5 торговых дней** из
   `bond_universe_history` (защита от однодневных выбросов).
2. Если истории < 5 дней (банк молодой) — считать по последнему снимку `bond_universe`, в ответе
   `basedOnDays: N` — UI покажет «оценка по N дн. данных».
3. Кэшировать результат в памяти на ~1 час (IMemoryCache или простое поле с таймстампом —
   посмотри, есть ли уже паттерн кэша в проекте; не изобретай сложное).

### C. API

1. `GET /api/analytics/relative-value` → для КАЖДОЙ сравнимой позиции портфеля (не floater/
   indexed/dataIncomplete): { positionId, basket: {sector, durationBucket, count, confidence},
   deviationFraction, percentile, verdict: Cheap|Fair|Rich (пороги: |deviation| < 20 б.п. — Fair,
   иначе по знаку; константы с doc-comment), basedOnDays } + `disclaimer` («относительная
   дешевизна к корзине по данным MOEX; НЕ оценка кредитного качества — большой спред может
   означать реальный риск эмитента»).
2. Для «дорогих» (Rich) позиций — топ-3 «дешёвых» (Cheap) кандидатов ИЗ ИХ ЖЕ корзины из банка
   (не скрытых, отсортированы по deviation убыв.): { secid, name, yieldFraction, deviationFraction,
   liquidityScore }.
3. Интеграционный тест: засеянная вселенная (2 сектора × 2 корзины, ≥5 бумаг) + позиция с
   заниженным спредом → verdict Rich + кандидаты из корзины; молодая история → basedOnDays честный.

### D. Frontend

1. **Бейдж RV на слабых звеньях** ([Recommendations.tsx](../bonds-web/src/pages/Recommendations.tsx),
   карточки WeakLinks): «дорогая: −38 б.п. к корзине» (оранжевый) / «дешёвая: +25 б.п.» (зелёный) /
   ничего при Fair; tooltip: корзина, перцентиль, confidence, basedOnDays.
2. **Новая секция «Дорогие бумаги — дешёвые соседи по корзине»** на странице рекомендаций (после
   «Слабых звеньев»): для каждой Rich-позиции — строка с её отклонением и 3 кандидатами-чипами
   (имя, YTM, +N б.п. к корзине, бейдж ликвидности); клик по кандидату — прокидывает его в
   сравнивалку задачи 27 (открыть MarketComparator с предвыбранной бумагой — посмотри её пропсы
   и сделай минимальную интеграцию, не перестраивая компонент).
3. **Таблица позиций** ([Positions.tsx](../bonds-web/src/pages/Positions.tsx)): компактный
   RV-индикатор в «Пометках» (маленький бейдж ↑дешёвая/↓дорогая с tooltip) — данные грузить лениво
   одним запросом, не блокируя таблицу; при ошибке — просто не показывать.
4. «?»-объяснение секции (паттерн ChartExplainIcon): что такое RV, почему это не «где YTM больше»,
   дисклеймер про кредитный риск.
5. Vitest: бейджи по verdict; секция с кандидатами; клик-прокидывание в сравнивалку; young-history
   подпись; отказ эндпоинта не роняет страницу/таблицу.

## Критерии приёмки

- [ ] Корзины консистентны с composition-бакетами; медианы по 5 дням истории с fallback и confidence.
- [ ] Verdict Cheap/Fair/Rich с порогами-константами; кандидаты — из корзины, не скрытые, с ликвидностью.
- [ ] UI: бейджи на слабых звеньях и в таблице, секция «дешёвых соседей», интеграция со сравнивалкой.
- [ ] Дисклеймер про «не кредитное качество» — на секции и в API.
