## Проблема (ТЗ владельца)

Блок 2 «Куда вложить сумму» сейчас смотрит **только на позиции портфеля** (+watchlist): пресет
«максимум доходности» = `GET /api/analytics/allocation` строит кандидатов из holdings
([AnalyticsEndpoints.cs:734-827](../src/Bonds.Api/Endpoints/AnalyticsEndpoints.cs)),
`CashAllocationService` документирован как «НЕ скринер по всей вселенной»
([CashAllocationService.cs:3-11](../src/Bonds.Core/Analytics/CashAllocationService.cs)). Владелец
хочет, чтобы пользователь мог выбирать **источник кандидатов**:
- **«весь рынок — доходные»**: все фикс-купонные бумаги рынка, самые доходные (флоатеры —
  исключены: по ним доходность несравнима, см. задачу 31);
- **«рекомендованные»**: рыночные кандидаты, отфильтрованные по позитивным риск-сигналам
  (ликвидность+листинг, неэкстремальный спред) + диверсификация — «более-менее надёжные аналоги»
  (это вариант «1+3» из обсуждения; не рейтинг агентств);
- **«мой портфель»**: текущее поведение (только позиции пользователя).

## Что сделать

### A. Параметр источника у аллокации

1. Расширить `GET /api/analytics/allocation` параметром `source={portfolio|market|recommended}`
   (дефолт `portfolio` — обратная совместимость). Параметр `includeWatchlist` осмыслен только для
   `portfolio` (для market/recommended игнорируется — вся вселенная и так шире).
2. Ответ `AllocationResponseDto` дополнить: у каждой `AllocationLineDto` — `RiskSignalsDto`
   (из задачи 33) для рыночных источников (для `portfolio` — можно тоже посчитать по банк-записи,
   если есть; иначе `null`). Эхо выбранного `source` в ответе.

### B. Построение рыночных кандидатов (`market`/`recommended`)

1. Источник — банк `bond_universe` через существующий репозиторий (как в `GET /api/universe`).
   Применить `UniverseHygieneFilter` (гигиенически-чистые), исключить `IsFloater == true`,
   исключить бумаги без `YieldFraction`/`DurationYears`/`PricePercent` (нечем ранжировать/оценить
   лот). Маппинг в `CashAllocationCandidate`:
   - `EffectiveYield = YieldFraction` (доля);
   - цена одного лота (рубли) = `PricePercent/100 × FaceValue` (+ при наличии — НКД; если банк не
     даёт НКД, документировать приближение «чистая цена», как делает конструктор);
   - `Issuer = null` на банк-слое → концентрация по эмитенту недоступна (см. B3);
   - `Sector` — из банка.
2. **`recommended`** = `market`, дополнительно отфильтрованный по риск-сигналам
   (`CandidateRiskSignalService`, задача 33): исключить кандидатов с `Liquidity == Caution` и со
   спредом `Caution` (спред заметно выше медианы корзины = повышенный риск). Порог/матрицу — те же,
   что в задаче 33. Оставшихся ранжировать по доходности убыв.
3. **Концентрация для рыночных источников — по СЕКТОРУ, не по эмитенту** (issuer в банке
   отсутствует). Ввести лимит доли сектора (напр. `DefaultMaxSectorSharePercent`, значение
   согласовать — предложи 30–40% и закрепи константой с обоснованием в doc-comment) в
   `CashAllocationService` как **альтернативную ось** концентрации: сервис должен уметь ограничивать
   либо по issuer (источник `portfolio`, как сейчас), либо по sector (рыночные источники). Не ломать
   существующую issuer-ветку. Плюс жёсткое правило «не больше 1 позиции на secid» (лот=1 бумага —
   допущение модели, см. BACKLOG техдолг). Отсечённые — в `AllocationSkipDto` с причиной.
   > **NEEDS DECISION (эскалировать оркестратору, если развилка блокирует):** значение
   > `DefaultMaxSectorSharePercent` и нужно ли для `market` (в отличие от `recommended`) вовсе
   > применять секторный лимит или только issuer-недоступность → тогда `market` = чистый greedy по
   > доходности + 1 лот/secid, а секторную диверсификацию оставить только `recommended`. Предложение
   > исполнителю: `market` — greedy по доходности + 1 лот/secid (без секторного лимита);
   > `recommended` — greedy + секторный лимит + фильтр риск-сигналов. Так «весь рынок» честно
   > показывает самое доходное, а «рекомендованные» — диверсифицированное и отфильтрованное.
4. Производительность: банк ~3400 строк, сортировка/фильтр in-memory — приемлемо (как
   `GET /api/universe`, см. техдолг). Ограничить итоговое число рассмотренных кандидатов разумным
   потолком (напр. топ-200 по доходности до жадного прохода), чтобы ответ был компактным; залогировать
   потолок в дисклеймере/ответе, если он режет выдачу (не «молча»).
5. Дисклеймер аллокации для рыночных источников обновить: «кандидаты — вся фикс-купонная вселенная
   банка (гигиенический фильтр), оценки по биржевой статистике, не инвестрекомендация».

### C. Единицы

- `EffectiveYield` — доля; цена лота — рубли (`PricePercent/100 × FaceValue`); секторная доля —
  проценты только если следуешь существующему `SharePercent`-исключению, иначе доля. Сверься с тем,
  как `CashAllocationService` уже считает issuer-долю (проценты `maxIssuerSharePercent`), и будь
  консистентен: секторный лимит — в тех же единицах, что issuer-лимит. Doc-comment на каждой новой
  границе.

### D. Тесты (эталонные числа руками)

Юнит (`CashAllocationService`):
1. Секторный лимит: кандидаты двух секторов, лимит 30% → распределение уважает лимит; эталон вручную.
2. Issuer-ветка (источник portfolio) не изменилась (регресс существующих тестов).
3. Флоатер-кандидат рыночного источника исключён (перекрытие с задачей 31, но проверить на market).
4. 1 лот на secid соблюдается.

Интеграционные:
5. `GET /api/analytics/allocation?source=market&amountRub=...` — кандидаты из банка, не из портфеля;
   флоатеров нет; риск-сигналы в строках; дисклеймер рыночный.
6. `source=recommended` — уже фикс-купонные, Caution-ликвидность/спред отфильтрованы.
7. `source=portfolio` (дефолт) — прежнее поведение (регресс).

## Критерии приёмки

- [ ] `GET /api/analytics/allocation` принимает `source`; `market`/`recommended` смотрят на всю
      фикс-купонную вселенную банка, `portfolio` = прежнее поведение (дефолт).
- [ ] Флоатеры не попадают в рыночную аллокацию; строки несут риск-сигналы.
- [ ] `recommended` отфильтрован по позитивным риск-сигналам + диверсифицирован; `market` показывает
      самое доходное (согласно принятому решению по B3).
- [ ] Концентрация рыночных источников — по сектору (issuer в банке отсутствует, задокументировано);
      1 лот на secid; отсечённые — в skip с причиной; потолок кандидатов не режет «молча».
- [ ] Единицы (доля/рубли/секторная доля) задокументированы и консистентны с issuer-веткой.
- [ ] Существующие тесты не ослаблены; `scripts/pre-push-check.sh --all` — зелёный.
