## Проблема (ТЗ владельца)

Сейчас страница «Рекомендации» ([bonds-web/src/pages/Recommendations.tsx](../bonds-web/src/pages/Recommendations.tsx))
— 5 секций, каждая решает сценарий частично: `WeakLinksSection` (слабые звенья),
`RelativeValueSection` (только Rich-позиции + дешёвые соседи), `ReplacementsSection` (матрица
замен), `BasketConstructor` (куда вложить), `WatchlistSection`. Целевая структура — **два главных
блока, один сценарий = один блок**:

- **Блок 1 «Слабые позиции → подобрать замену»**: список самых слабых по доходности позиций; у
  каждой кнопка «подобрать замену» → выпадашка с режимами: **«самые доходные рынка»** / **«дешёвые
  соседи по корзине (RV)»** / **«поиск»**. Один блок вместо нынешних «слабые звенья» + «матрица
  замен» + «RV-секции». Кандидаты ранжируются по доходности; у каждого — **риск-сигналы** (два
  тега: ликвидность+листинг, спред) как позитивный/негативный индикатор, **явно не** рейтинг
  агентств (владелец: в будущем добавим отдельный тег рейтинга).
- **Блок 2 «Куда вложить сумму»**: конструктор остаётся, но с **переключателем источника
  кандидатов**: «весь рынок — доходные» / «рекомендованные» / «мой портфель» (последнее = как
  раньше). Кандидаты рыночных источников — с теми же риск-сигналами.

## Что сделать

### A. Типы (зеркало DTO задач 33/34)

1. В [api/types.ts](../bonds-web/src/api/types.ts) добавить:
   `RiskSignalsResponse` (поля `liquidity: 'Good'|'Neutral'|'Caution'`, `liquidityLabel`,
   `spread: 'Good'|'Neutral'|'Caution'`, `gspreadFraction`, `spreadVsBasketMedianFraction`),
   `ReplacementCandidate`, `ReplacementCandidatesResponse` (mode, positionIsin, candidates,
   disclaimer). Расширить `AllocationLine`/`AllocationResponse` полем `riskSignals?` и эхо `source`.
   Сверяй строковые enum-значения с сериализацией бэкенда буквально.
2. API-обёртки в [api/recommendations.ts](../bonds-web/src/api/recommendations.ts):
   `fetchReplacementCandidates(positionId, mode, limit?)`,
   `fetchAllocation(amountRub, { source, includeWatchlist? })` (расширить существующую).

### B. Блок 1 — «Слабые позиции → подобрать замену»

1. Переиспользовать существующий отбор слабых позиций (`buildSellCandidates`
   [useRecommendationsStore.ts:39-77](../bonds-web/src/store/useRecommendationsStore.ts)) и разбивку
   «вне сравнения» (там же). После fix `yieldKind` (задача 32) флоатеры-позиции уходят в «вне
   сравнения» корректно.
2. Компонент общий риск-тег — новый `components/RiskSignalBadges.tsx`: два бейджа (ликвидность+
   листинг, спред) с цветом по уровню (Good — зелёный, Neutral — серый, Caution — оранжевый) и
   `Tooltip`/`ChartExplainIcon`-подсказкой; **подпись-дисклеймер** «сигналы по биржевой статистике,
   не рейтинг агентств». Переиспользуется в блоке 1 и блоке 2.
3. Карточка слабой позиции: кнопка/раскрывашка «подобрать замену» → `SegmentedControl`/`Tabs` с
   режимами **«доходные рынка» / «дешёвые соседи (RV)» / «поиск»**:
   - «доходные рынка» и «дешёвые соседи» → `fetchReplacementCandidates(positionId, 'market'|'rv')`,
     список кандидатов с доходностью/дюрацией + `RiskSignalBadges`; клик по кандидату →
     существующий путь выгоды (`postReplacement` → карточка `ReplacementBreakdown`).
   - «поиск» → существующий `MarketComparator` (после задачи 32 флоатеры из его выпадашки убраны),
     дополнить риск-сигналами из ответа `POST /replacement` (задача 33 A4).
4. Убрать отдельные секции `RelativeValueSection` и `ReplacementsSection` со страницы (их сценарий
   поглощён блоком 1). RV-бейдж на карточке (`RelativeValueBadge`) можно сохранить как индикатор
   «дорогая/дешёвая» позиции — на усмотрение, но без отдельной большой секции. Матрицу замен
   (`ReplacementsSection`/`ReplacementPairCard`/`RejectedPairsTable`) удалить со страницы; если
   какой-то её элемент ценен — перенести внутрь блока 1, не оставлять параллельной секцией.
   > Компоненты `RelativeValueSection.tsx`/`ReplacementsSection`-части можно удалить, если больше
   > нигде не используются (проверь ссылки). Не удаляй `MarketComparator`, `ReplacementBreakdown`,
   > `RelativeValueBadge` — они переиспользуются.
5. Состояния загрузки/ошибки/пустоты для каждого режима; отказ `replacement-candidates` не роняет
   блок и страницу.

### C. Блок 2 — «Куда вложить сумму» с источником кандидатов

1. В `BasketConstructor` ([bonds-web/src/components/BasketConstructor.tsx](../bonds-web/src/components/BasketConstructor.tsx))
   добавить переключатель источника (`SegmentedControl`): **«Весь рынок — доходные»** (`market`) /
   **«Рекомендованные»** (`recommended`) / **«Мой портфель»** (`portfolio`). Пресет «Максимум
   доходности» (`handleUseGreedyPreset` [:193-211](../bonds-web/src/components/BasketConstructor.tsx))
   вызывает `fetchAllocation(amountRub, { source })` с выбранным источником (дефолт — сохранить
   прежний UX: если сомневаешься, дефолт `market`, но не ломай возможность вернуться к `portfolio`).
2. Показать риск-сигналы (`RiskSignalBadges`) в строках результата аллокации/предпросмотра
   кандидатов. Пояснить режимы «?»-подсказкой: «весь рынок — самые доходные фикс-купонные;
   рекомендованные — отфильтрованы по ликвидности/спреду и диверсифицированы; мой портфель — только
   ваши позиции». Дисклеймер из ответа бэкенда — показать.
3. Ручное добавление строки (существующий Select с `fetchUniverse`) сохранить; расчёт корзины
   (`POST /basket`) не менять.

### D. Композиция страницы и сторы

1. `Recommendations.tsx`: две главные секции (Блок 1, Блок 2) + дисклеймер вверху. `WatchlistSection`
   — оставить как вспомогательную секцию внизу (отдельный стор, независимая загрузка) ИЛИ свернуть в
   аккордеон; не превращать в третий «главный» блок. Порядок: Блок 1 → Блок 2 → (watchlist).
2. Сохранить независимую загрузку: блок 1 и блок 2 грузятся своими путями, отказ RV/candidates не
   роняет страницу (паттерн `loadRelativeValue` отдельно от `load` уже есть — переиспользовать
   принцип). Пересмотреть `useRecommendationsStore.load()` (сейчас грузит comparison+composition+
   matrix одним `Promise.all`): матрица замен как секция удаляется — убрать лишний запрос
   `fetchReplacementMatrix`, если он больше не нужен (проверь, что `bestPairs`/`rejectedPairs` нигде
   не читаются после переработки).
3. Мобильная адаптация: карточки (`PositionCard`-паттерн + `useMediaQuery`), бейджи переносятся.

### E. Vitest-кейсы

1. Блок 1: у слабой позиции переключение режимов дёргает правильный эндпоинт; кандидаты
   рендерятся с двумя риск-бейджами; выбор кандидата открывает карточку выгоды.
2. Блок 1: пустой/ошибочный ответ режима не роняет блок; «вне сравнения» позиции (флоатеры)
   показаны отдельно.
3. Блок 2: переключение источника меняет запрос `fetchAllocation` (`source`); строки несут
   риск-бейджи; дисклеймер виден.
4. Страница: старые секции (матрица замен, отдельная RV-секция) отсутствуют; дисклеймер вверху есть.
5. Регресс: удалённые компоненты не импортируются нигде (иначе typecheck упадёт — гарантия чистоты).

## Критерии приёмки

- [ ] Страница = два главных блока по ТЗ; параллельных секций «матрица замен»/«RV» больше нет.
- [ ] Блок 1: слабая позиция → выпадашка режимов (доходные рынка / дешёвые соседи RV / поиск),
      кандидаты с двумя риск-сигналами (не выдаются за рейтинг агентств), выбор → карточка выгоды.
- [ ] Блок 2: конструктор с переключателем источника (весь рынок / рекомендованные / портфель),
      рыночные кандидаты с риск-сигналами и дисклеймером.
- [ ] Независимая загрузка виджетов сохранена; мобильная версия и `AppLayout`/роут не сломаны;
      дисклеймеры на месте.

      --all` — зелёный.
