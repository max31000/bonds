## Проблема (подтверждена live-данными MOEX, найдена владельцем в проде)

Брус 2Р06 (`RU000A10EC63`): фикс-купон 22.75% (BONDTYPE «Биржевая облигация», COUPONPERCENT
задан), put-оферта 2028-02-18. MOEX bondization отдаёт 24 известных купона до оферты и 12
**неизвестных после** (ставка будет пересмотрена на оферте — стандарт для put-оферт).

Правило классификации [BondSyncService.cs:393](../src/Bonds.Infrastructure/Sync/BondSyncService.cs):
```csharp
var hasFloatingCoupon = bondization.Coupons.Any(c => !c.IsKnown) || (info?.LooksLikeFloater ?? false);
```
`Coupons.Any(!IsKnown)` истинно для ЛЮБОЙ фикс-бумаги с офертой и пересмотром ставки → ложный
`CouponType.Floating`. Каскад: 422 в `POST /analytics/replacement` для такой цели (владелец
упёрся именно в это); собственные позиции с офертой уходят в «вне сравнения» слабых звеньев
(`YieldKind=CurrentYield`), исключаются из матрицы замен, RV-вердиктов и (после T-31) из
аллокации по портфелю. При этом движок умеет считать доходность к оферте: горизонт режется
`OfferCutoffResolver.Resolve(asOf, maturityDate, offers)`
([BondMetricsCalculator.cs:46,100](../src/Bonds.Core/Calculation/BondMetricsCalculator.cs)),
`CalculatedToOffer` уже есть в DTO. Т.е. неизвестные купоны **после ближайшей оферты вообще не
участвуют** в расчёте доходности такой бумаги — и не должны влиять на классификацию.

## Что сделать

### A. Правило классификации

1. В `EnrichFromMoexAsync` ([BondSyncService.cs:385-399](../src/Bonds.Infrastructure/Sync/BondSyncService.cs))
   заменить правило на: бумага — флоатер, если есть неизвестный купон **с датой не позже
   ближайшей будущей оферты**; неизвестные купоны строго после ближайшей оферты — норма
   пересмотра ставки, НЕ флоатер. Формально:
   - `nearestOffer` = минимальная будущая дата из `bondization.Offers` (сегодня по UTC/MSK —
     сверься, как берёт дату `OfferCutoffResolver`, и переиспользуй его резолв горизонта, если
     сигнатура позволяет — не дублируй логику «ближайшей оферты» третий раз);
   - оферты нет → прежнее поведение: любой `!IsKnown` купон = Floating;
   - оферта есть → Floating только если `Coupons.Any(c => !c.IsKnown && c.<дата купона> <= nearestOffer)`
     (имя поля даты возьми из фактической модели купона);
   - `info?.LooksLikeFloater` (BONDTYPE «Флоатер» / отсутствие COUPONPERCENT) — оставить в OR
     как было: для настоящих флоатеров с известными ближними купонами это единственный сигнал.
2. Вынести правило в **чистую тестируемую функцию** (напр. `internal static bool
   HasFloatingCoupon(IReadOnlyList<...> coupons, IReadOnlyList<...> offers, DateOnly asOf, bool
   looksLikeFloater)` — рядом в `BondSyncService` или в Core, по месту виднее), а в
   `EnrichFromMoexAsync` — вызов. Doc-comment с примером Брус 2Р06 и ссылкой на этот план.
3. Проверить и починить, если нужно, **ту же ошибку в других местах**: grep по
   `Any(c => !c.IsKnown)` / `!c.IsKnown` по всему `src/` — если где-то ещё из этого выводится
   «флоатерность» (не «неполнота данных»!), применить то же правило. `DataIncomplete`
   (дырки между известными купонами, парсер) — НЕ трогать.

### B. Самовосстановление данных

Ничего писать не нужно — только проверить и зафиксировать в отчёте: `EnrichFromMoexAsync`
вызывается для каждой позиции при каждом синке ([BondSyncService.cs:140](../src/Bonds.Infrastructure/Sync/BondSyncService.cs))
и через `EnrichByIsinAsync`/materialize для watchlist и кандидатов банка — т.е. `CouponType`
затронутых бумаг пересчитается сам при первом обращении после деплоя.

### C. Тесты (юнит, эталоны руками; сеть не нужна — синтетические расписания)

На вынесенную чистую функцию:
1. Фикс-бумага с офертой, все купоны до оферты включительно известны, после — неизвестны
   (кейс Брус 2Р06) → **Fixed**.
2. Неизвестный купон ДО ближайшей оферты → Floating.
3. Неизвестный купон с датой = дате оферты → Floating (граница: купон на дату оферты ещё
   обязан быть известен).
4. Без оферт + неизвестный купон → Floating (прежнее поведение, регресс).
5. Все купоны известны, оферта есть → Fixed.
6. `looksLikeFloater=true` при полностью известных купонах → Floating (BONDTYPE-сигнал жив).
7. Оферта в прошлом (уже прошла), неизвестные купоны после неё → трактуется как «оферты нет»
   → Floating (прошедшая оферта не оправдывает неизвестные купоны).

Плюс: существующие тесты синка/классификации не ослаблены.
